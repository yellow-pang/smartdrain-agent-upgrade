## 4. 프론트엔드 기술 선정근거

### 4.1 Next.js와 React

Next.js는 React 기반 프레임워크로, 대시보드와 상세 화면을 라우팅 구조로 나누기 쉽다. 본 프로젝트는 `/dashboard`, `/drains/[drainId]`처럼 화면 단위가 명확하므로 Next.js가 적합하다.

| 선정 이유 | 설명 |
|---|---|
| 화면 구조 명확화 | 대시보드와 상세 페이지를 분리하기 쉽다. |
| 컴포넌트 재사용 | 지도 패널, 위험 시설 카드, 상세 정보 카드를 재사용할 수 있다. |
| 배포 편의성 | Vercel 배포와 잘 맞는다. |
| 확장성 | 관리자 인증, 작업자 화면, 운영 화면 추가 시 라우팅 확장이 가능하다. |

### 4.2 TypeScript

TypeScript는 빗물받이 상태, 센서 데이터, 위험도, 지도 좌표, WebSocket 이벤트 구조를 명확하게 관리하기 위해 사용한다.

```ts
type DrainStatus = "good" | "caution" | "danger" | "unknown";

type Drain = {
  id: string;
  code: string;
  address: string;
  status: DrainStatus;
  obstructionRatio: number;
  waterHeight: number;
  flowVelocity: number;
  riskScore: number;
  updatedAt: string;
  latitude: number;
  longitude: number;
};
```

| 화면 표시 | 코드값 | 설명 |
|---|---|---|
| 양호 | `good` | 배수 상태가 안정적이며 위험도가 낮은 상태 |
| 주의 | `caution` | 일부 막힘 또는 센서 수치 상승으로 모니터링이 필요한 상태 |
| 위험 | `danger` | 침수 가능성이 높아 즉시 확인이 필요한 상태 |
| 판단불가 | `unknown` | 이미지 품질 저하, 센서 누락, 모델 신뢰도 부족 등으로 판단이 어려운 상태 |

### 4.3 Tailwind CSS와 shadcn/ui

Tailwind CSS는 대시보드 UI를 빠르게 구성하기 위해 사용한다. shadcn/ui는 카드, 버튼, 배지, 탭 등 반복 UI 구성 요소를 빠르게 만들기 위해 사용한다.

| 적용 UI | 사용 이유 |
|---|---|
| Card | 대시보드 요약 정보, 상세 정보 영역 구성 |
| Badge | 양호, 주의, 위험, 판단불가 상태 표시 |
| Button | 상세 보기, 새로고침, 고도화 기능 진입 버튼 구성 |
| Tabs | 상세 정보, 이력, 센서 데이터 화면 구분 |
| Progress | 막힘 정도, 수위 비율 표현 |

### 4.4 Axios와 TanStack Query

Axios는 FastAPI와 REST API 통신을 수행하고, TanStack Query는 API 응답의 로딩, 에러, 캐싱, 재조회 상태를 관리한다.

| 기술 | 역할 |
|---|---|
| Axios | API 요청 수행 |
| TanStack Query | 서버 상태, 캐싱, 로딩, 에러 관리 |

주요 API는 다음과 같다.

```text
GET /api/drains
GET /api/drains/{drain_id}
GET /api/drains/{drain_id}/risk-history
GET /api/dashboard/summary
```

### 4.5 Zustand

Zustand는 전역 상태를 관리하기 위해 사용한다. WebSocket으로 수신한 위험도 변경 이벤트를 지도 마커, 위험 시설 목록, 상세 화면에 동시에 반영해야 하므로 전역 상태 관리가 필요하다.

| 관리 대상 | 설명 |
|---|---|
| 선택된 빗물받이 ID | 지도 마커 또는 목록 클릭 시 선택된 시설 관리 |
| 빗물받이 최신 상태 | WebSocket 이벤트로 갱신되는 위험도 상태 관리 |
| WebSocket 연결 상태 | connected, disconnected 등 연결 상태 관리 |
| 위험 이벤트 | 위험 발생 이벤트와 화면 표시 상태 관리 |

### 4.6 Kakao Maps API

Kakao Maps API는 국내 지도 기반 위치 표시를 위해 선정하였다. 위도와 경도를 기반으로 빗물받이 마커를 표시하고, 위험도에 따라 마커 색상을 다르게 표현할 수 있다.

### 4.7 Recharts

Recharts는 상세 화면의 수위, 유속, 위험도 변화 이력을 시각화하기 위해 사용한다.

| 적용 데이터 | 설명 |
|---|---|
| 수위 변화 | 침수 위험 판단 근거 시각화 |
| 유속 변화 | 배수 흐름 상태 확인 |
| 위험도 점수 변화 | XGBoost 판단 결과 이력 확인 |
| 최근 24시간 또는 7일 이력 | 시간 흐름에 따른 상태 변화 확인 |

## 5. 백엔드 기술 선정근거

### 5.1 FastAPI

FastAPI는 Python 기반 API 서버 개발을 위해 선정하였다. 본 프로젝트는 OpenCV, YOLO, XGBoost 등 Python 기반 AI 분석 기술과 연동해야 하므로 백엔드 역시 Python 기반이 적합하다. FastAPI는 REST API와 WebSocket을 모두 지원한다.

| 선정 이유 | 설명 |
|---|---|
| Python 기반 AI 연동 | OpenCV, YOLO, XGBoost와 연동하기 쉽다. |
| API 개발 속도 | REST API를 빠르게 구현할 수 있다. |
| WebSocket 지원 | 위험도 변경 이벤트를 실시간 전달할 수 있다. |
| 자동 문서화 | Swagger UI 기반 API 문서 확인이 가능하다. |
| 유지보수성 | 기능별 라우터 분리와 모듈화가 쉽다. |

### 5.2 Uvicorn

Uvicorn은 FastAPI 애플리케이션 실행을 위한 ASGI 서버로 사용한다.

```bash
uvicorn main:app --reload
```

### 5.3 Pydantic

Pydantic은 API 요청 및 응답 데이터의 구조를 검증하기 위해 사용한다.

```python
class SensorDataCreate(BaseModel):
    drain_id: int
    water_level_cm: float
    flow_velocity_mps: float
    measured_at: datetime
```

## 6. 데이터베이스 기술 선정근거

### 6.1 PostgreSQL

PostgreSQL은 빗물받이 정보, 센서 데이터, YOLO 분석 결과, XGBoost 판단 결과를 구조적으로 저장하기 위해 선정한다.

| `yolo_result_data` | 이미지 URL, 막힘 정도, confidence score |
| `xgboost_data` | 최종 위험도, 위험 점수, 판단 시간 |

### 6.2 SQLAlchemy, Alembic, psycopg

| 기술 | 역할 |
|---|---|
| SQLAlchemy | Python 코드로 DB 테이블을 다루는 ORM |
| Alembic | DB 구조 변경 이력 관리 |
| psycopg | Python과 PostgreSQL 연결 드라이버 |

### 6.3 Docker Compose 기반 PostgreSQL 실행

팀원 간 DB 설치 환경 차이를 줄이기 위해 Docker Compose 기반 PostgreSQL 실행을 권장한다.

```yaml
services:
  postgres:
    image: postgres:16
    container_name: flood_postgres
    ports:
      - "5432:5432"
    environment:
      POSTGRES_DB: flood_db
      POSTGRES_USER: flood_user
      POSTGRES_PASSWORD: flood_password
    volumes:
      - postgres_data:/var/lib/postgresql/data

volumes:
  postgres_data:
```

## 7. AI 분석 기술 선정근거

### 7.1 OpenCV

OpenCV는 CCTV 스냅샷 또는 샘플 이미지를 YOLO가 분석하기 쉬운 형태로 변환하기 위해 사용한다.

| 역할 | 설명 |
|---|---|
| 이미지 읽기 | URL 또는 저장 경로 기반 이미지 로드 |
| 전처리 | 리사이징, 밝기 보정, 노이즈 제거, ROI 추출 |
| 분석 입력 생성 | YOLO 모델이 사용할 이미지 데이터 준비 |

### 7.2 YOLO

YOLO는 이미지에서 빗물받이와 이물질 상태를 분석하기 위해 사용한다. YOLO 결과는 XGBoost 최종 위험도 판단의 입력값으로 활용된다.

| 출력값 | 설명 |
|---|---|
| `obstruction_ratio` | 이미지 기반 막힘 비율 |
| `confidence_score` | 이미지 분석 신뢰도 |
| `yolo_status` | 이미지 기반 상태 코드 |

YOLO 실행 방식은 구현 상황에 따라 백엔드 내부 실행 또는 분석 결과 연동 방식 중 하나로 선택할 수 있다. MVP 문서에서는 실행 방식보다 YOLO 결과가 XGBoost 판단에 사용되는 구조를 기준으로 정의한다.

### 7.3 XGBoost

XGBoost는 MVP 필수 기술이다. YOLO 이미지 분석 결과와 수위·유속 센서 데이터를 종합하여 최종 위험도를 판단한다.

| 입력값 | 설명 |
|---|---|
| `obstruction_ratio` | YOLO 기반 막힘 비율 |
| `confidence_score` | YOLO 분석 신뢰도 |
| `water_level_cm` | 수위 값 |
| `flow_velocity_mps` | 유속 값 |

| 출력값 | 설명 |
|---|---|
| `risk_score` | 수치화된 침수 위험 점수 |
| `risk_level` | `good / caution / danger / unknown` 중 하나 |
| `final_decision` | 관리자 화면에 표시할 최종 판단 코드 |

MVP 단계에서는 실제 운영 데이터가 충분하지 않을 수 있으므로 샘플 이미지와 모의 센서 데이터를 기반으로 한 PoC 수준의 위험도 판단 모델로 적용한다. 향후 실제 데이터가 축적되면 학습 데이터 품질을 개선하고 판단 정확도를 고도화한다.

## 8. 실시간 처리 기술 선정근거

### 8.1 WebSocket

WebSocket은 MVP 필수 실시간 처리 기술이다. 위험도 변경 이벤트를 프론트엔드에 전달하여 관리자가 새로고침 없이 지도 마커, 위험 시설 목록, 상세 화면의 상태 변화를 확인할 수 있도록 한다.

| 적용 위치 | 설명 |
|---|---|
| 지도 마커 | 위험도 변경 시 마커 색상 갱신 |
| 위험 시설 목록 | 위험 시설 목록 정렬 및 상태 변경 |
| 상세 화면 | 현재 수위, 유속, 위험도 상태 반영 |
| 요약 카드 | 양호, 주의, 위험, 판단불가 개수 갱신 |

```json
{
  "type": "DRAIN_STATUS_UPDATED",
  "payload": {
    "drainId": "DR-004",
    "riskCode": "danger",
    "riskScore": 0.91,
    "waterHeight": 85,
    "flowVelocity": 0.05,
    "obstructionRatio": 0.88,
    "updatedAt": "2026-06-17T10:30:00"
  }
}
```

## 12. 결론

본 프로젝트의 기술스택은 MVP 구현 가능성, AI 분석 연동성, 실시간 관제 기능, 팀 협업 환경을 기준으로 선정하였다.

프론트엔드는 Next.js 기반으로 지도 대시보드와 상세 화면을 구성하고, Kakao Maps API와 Recharts를 활용해 위치 정보와 위험도 변화를 시각화한다. WebSocket을 활용하여 위험도 변경 이벤트를 실시간으로 반영한다.

백엔드는 FastAPI를 중심으로 구성하며, OpenCV, YOLO, XGBoost 등 Python 기반 AI 분석 흐름과 연동한다. 데이터베이스는 PostgreSQL을 우선 선정하고 Docker Compose 기반으로 실행하여 팀원 간 개발 환경을 통일한다.

MVP에서는 XGBoost와 WebSocket을 핵심 기술로 포함하며, 실제 CCTV, 실제 센서, 대응 요청, 외부 알림, LLM 요약은 고도화 기술로 분리한다.
