# 16 테스트 전략 및 E2E 검증

> 문서 상태: 현재 기준 문서 · 자동화 여부와 실행 결과를 구분한다.

## 1. 자동화 현황

| 범위 | 현재 상태 | 근거 |
| --- | --- | --- |
| AI unit·contract | 코드 확인 | `ai_service/**/tests` |
| Backend API | 미구현 | backend pytest 파일 미확인 |
| Frontend component | 미구현 | test runner 설정 미확인 |
| WebSocket·브라우저 E2E | 미구현 | 수동 통합 문서만 존재 |
| Compose smoke | 절차 코드 확인 | Jenkins smoke script·Compose 설정 |
| 런타임 자동 시뮬레이터 | 구현 완료 / 자동 E2E 미구축 | `GET/POST /api/realtime-simulator/*` + 대시보드 제어 UI |

## 2. 대표 E2E 시나리오

`분석 요청 → AnalysisJob 생성 → AI accepted → YOLO 분석/callback → XGBoost 분석/callback → DB 저장 → Drain 상태 갱신 → WebSocket → Frontend Query cache·상태 갱신 → 지도·목록·상세 반영`

성공 검증은 API 응답, `analysis_jobs` 상태, 결과 FK 연결, WebSocket 세 이벤트, 화면 반영을 각각 확인한다. 수동 확인 결과는 날짜·환경·입력 drain code를 함께 기록한다.

자동 시뮬레이터 시나리오는 `status -> start -> status(running=true) -> tick 반영 -> stop -> status(running=false)` 순서로 확인한다. 자동 모드에서도 수동 `POST /api/analysis/async-run` 흐름이 회귀하지 않아야 한다.

## 3. 실패·경계 시나리오

| 시나리오 | 기대 확인 |
| --- | --- |
| AI 연결 실패·timeout | job이 `failed`와 오류 내용을 남기는지 |
| YOLO/XGBoost callback 중복 | DB 중복 저장·재broadcast가 없는지 |
| XGBoost 선도착 | 409 및 `xgboost_received_without_yolo` 상태인지 |
| 최신 센서 없음 | 분석 시작이 404인지 |
| WebSocket 단절·재연결 | Frontend가 Query cache를 무효화하는지 |
| 모델 파일 없음 | ai-service가 기동 실패하는지 |
| DB·Nginx 오류 | health·proxy·로그에서 원인을 식별할 수 있는지 |

## 4. 실행 기준

실행 가능한 경우 `python -m pytest ai_service -q`, `npm --prefix frontend run lint`, `npm --prefix frontend run build`, `docker compose config --quiet`를 수행한다. 모델·환경변수·Docker가 준비되지 않은 항목은 `미실행`으로 기록한다. MVP 완료 판단에는 최소 AI test, Compose config, 분석 callback 수동 흐름, Nginx REST·WebSocket smoke를 포함한다.
