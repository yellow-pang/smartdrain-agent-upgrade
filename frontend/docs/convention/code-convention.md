# Frontend Code Convention

SmartDrain frontend 코드 컨벤션을 정리하기 위한 문서입니다. 팀 공통 컨벤션 중 frontend 코드 작성에 직접 영향을 주는 내용을 우선 반영하고, 세부 기준은 추후 Notion 내용을 옮겨 보완합니다.

## 1. 목적

frontend 코드 작성 기준을 한곳에 모아 Codex와 개발자가 같은 기준으로 작업하기 위한 문서입니다.

코드 스타일, 파일 배치, 컴포넌트 작성 방식, mock data와 API 데이터 구분 기준을 일관되게 유지하는 것을 목표로 합니다.

## 2. 적용 범위

이 문서는 `/frontend` 내부의 TypeScript, React, Next.js, Tailwind CSS 코드에 적용합니다.

브랜치 전략, 커밋 메시지, PR 규칙은 코드 자체보다 협업 문서 성격이 강하므로 `documentation-convention.md`와 `AGENTS.md`에서 주로 다룹니다.

## 3. 네이밍 규칙

변수, 함수, 컴포넌트, 타입, 파일, 폴더 이름은 이름만 보아도 역할을 알 수 있게 작성합니다.

- 컴포넌트 export 이름은 PascalCase를 사용합니다.
- 컴포넌트 파일명은 `risk-map.tsx`, `sensor-trend-chart.tsx`처럼 lowercase kebab-case를 사용합니다.
- 역할이 모호한 축약어는 피하고, 화면이나 데이터의 의미가 드러나는 이름을 우선합니다.

## 4. 함수 작성 규칙

함수는 하나의 역할을 중심으로 작성하고, 입력값과 반환값의 의미가 드러나도록 타입을 명확히 둡니다.

반복되는 계산, 포맷팅, 상태 판정 로직은 컴포넌트 안에 길게 두기보다 필요한 경우 `lib/` helper로 분리합니다.

## 5. 컴포넌트 작성 규칙

React 컴포넌트는 함수형 컴포넌트를 기본으로 사용합니다.

화면 전체를 구성하는 route/page 컴포넌트와 재사용 가능한 UI 컴포넌트의 책임을 분리합니다. props는 필요한 값만 전달하고, 컴포넌트 내부에서 불필요하게 여러 관심사를 섞지 않습니다.

## 6. TypeScript 작성 규칙

TypeScript는 strict type checking을 전제로 작성합니다.

데이터 구조, API 응답, mock data는 가능한 한 타입을 명시해 실제 화면에서 사용하는 값의 형태가 드러나게 합니다. `any` 사용은 피하고, 예외가 필요한 경우 이유를 남깁니다.

## 7. Tailwind CSS 작성 규칙

Tailwind CSS utility를 기본 스타일 작성 방식으로 사용합니다.

- 상태 색상은 정상 `green`, 주의 `yellow` 또는 `amber`, 위험 `red` 계열을 일관되게 사용합니다.
- 반응형 UI는 모바일부터 데스크톱까지 텍스트 겹침과 레이아웃 깨짐이 없도록 확인합니다.
- 반복되는 class 조합이 많아지면 컴포넌트 분리나 helper 사용을 검토합니다.

## 8. shadcn 스타일 UI primitive 사용 기준

버튼, 카드, 배지, 탭 등 공통 UI는 먼저 `components/ui/`의 shadcn 스타일 primitive 사용 가능 여부를 확인합니다.

새 UI를 만들 때는 기존 primitive를 조합하는 방식을 우선하고, primitive 자체를 바꿀 때는 다른 화면에 미치는 영향을 확인합니다.

## 9. mock data와 실제 API 데이터 구분 기준

mock data와 실제 API 연동 데이터가 섞이지 않도록 파일 위치, 타입 이름, 사용 목적을 구분합니다.

mock data는 화면 검증과 데모를 위한 임시 데이터입니다.  
실제 API 응답 형식은 백엔드 문서, API 명세, 또는 팀 합의 내용을 우선합니다.

따라서 frontend에서 사용하는 mock data 구조를 실제 API 응답 형식으로 확정하지 않습니다.  
API 응답 타입이 필요한 경우에도 임시 타입 또는 참고 타입임을 명확히 표시합니다.

### mock data 작성 기준

- mock data는 화면 구성과 UI 상태 확인을 위한 데이터입니다.
- mock data는 실제 서버 응답과 동일하다고 가정하지 않습니다.
- mock data 파일에는 데모 데이터라는 의미가 드러나도록 작성합니다.
- mock data를 수정할 때는 어떤 화면 또는 컴포넌트에 영향을 주는지 확인합니다.
- mock data 구조를 크게 바꾸는 경우에는 사용자 확인 후 진행합니다.

예시:

```ts
// 화면 검증용 mock data 예시
const mockDrainStatusList = [
  {
    id: "drain-001",
    name: "강남역 1번 출구 빗물받이",
    riskLevel: "danger",
    waterLevel: 82,
  },
];
```

### 실제 API 데이터 작성 기준

실제 API 연동 시에는 백엔드 API 명세를 우선합니다.

API 응답 형식, 필드명, 에러 응답 구조, 상태 코드는 frontend에서 임의로 확정하지 않습니다.  
백엔드 명세가 확정되면 그 기준에 맞춰 타입, mock data, 화면 표시 로직을 함께 수정합니다.

API 연동을 준비할 때는 RESTful API 형태를 참고할 수 있습니다.

```text
GET    /domain
POST   /domain
GET    /domain/:id
PATCH  /domain/:id
DELETE /domain/:id
```

단, 위 경로 형식은 참고 예시이며 실제 엔드포인트는 백엔드 API 명세를 따릅니다.

### 임시 API 응답 타입 기준

백엔드 응답 형식이 확정되기 전까지 frontend에서 임시 타입이 필요한 경우, 성공/실패를 구분할 수 있는 형태를 참고합니다.

아래 타입은 확정 규칙이 아니라 참고 예시입니다.  
실제 API 응답 형식이 확정되면 백엔드 명세에 맞게 수정합니다.

```ts
type ApiSuccessResponse<T> = {
  success: true;
  data: T;
  message?: string;
};

type ApiErrorResponse = {
  success: false;
  message: string;
  error?: string;
};

type ApiResponse<T> = ApiSuccessResponse<T> | ApiErrorResponse;
```

이 형태를 참고하는 이유는 성공 응답과 실패 응답을 화면에서 구분하기 쉽기 때문입니다.

```ts
if (response.success) {
  // 성공 응답에서는 response.data를 사용합니다.
} else {
  // 실패 응답에서는 response.message를 사용자에게 보여줄 수 있습니다.
}
```

### API 응답 타입 사용 시 주의사항

- 백엔드 API 명세가 있으면 그 명세를 우선합니다.
- mock data 타입과 실제 API 응답 타입을 혼동하지 않습니다.
- 화면 컴포넌트에서 API 응답 전체를 직접 다루지 않습니다.
- 필요한 경우 `lib/` 또는 API helper에서 화면에 필요한 데이터 형태로 변환합니다.
- API 응답 형식이 바뀌면 관련 타입, mock data, 화면 표시 로직을 함께 확인합니다.
- `statusCode`, `timestamp`, `path`, `error` 같은 필드는 실제 백엔드 응답에 포함되는지 확인한 뒤 사용합니다.
- 응답 형식이 확정되지 않은 상태에서 특정 응답 구조를 프로젝트 표준처럼 문서화하지 않습니다.

## 10. 시간 복잡도와 공간 복잡도를 고려한 코드 작성 기준

데이터 목록, 차트, 지도 UI에서 불필요한 반복 계산이나 큰 메모리 사용을 피합니다.

렌더링마다 같은 배열을 반복 가공하거나, 큰 mock data를 화면 컴포넌트 내부에서 매번 새로 만드는 방식은 주의합니다.

## 11. 유지보수성을 고려한 코드 작성 기준

기능 변경 시 영향 범위를 줄이고, 다음 개발자가 이해하기 쉬운 구조를 우선합니다.

기능 변경과 전체 포맷팅을 한 작업에 섞지 않습니다. 중복 제거와 컴포넌트 분리는 실제 유지보수 비용을 줄일 때 진행합니다.

## 12. 가독성과 성능 최적화의 균형

성능 개선이 필요해도 코드 가독성을 크게 떨어뜨리는 변경은 신중히 진행합니다.

최적화가 필요한 경우 어떤 문제가 있었고, 변경 후 어떤 효과를 기대하는지 작업 기록에 남깁니다.

## 13. 금지하거나 주의할 코드 작성 방식

프로젝트에서 피해야 할 패턴과 리뷰에서 특히 확인할 항목입니다.

- ESLint 경고를 이유 없이 방치하지 않습니다.
- `any`를 습관적으로 사용하지 않습니다.
- mock data를 실제 API 응답처럼 확정된 구조로 오해하게 만들지 않습니다.
- 상태 색상 기준을 화면마다 다르게 쓰지 않습니다.
- 사용자 승인 없이 패키지, 라우팅 구조, 상태 관리 방식, 지도/차트 라이브러리를 바꾸지 않습니다.

## 14. 코드 스타일 도구와 포맷팅

팀 컨벤션 기준으로 ESLint와 Prettier 사용을 전제로 합니다.

- ESLint를 적용합니다.
- Prettier 사용을 전제로 하되, 현재 프로젝트 설정 파일 유무를 먼저 확인합니다.
- 세미콜론을 사용합니다.
- 문자열은 싱글 쿼트를 사용합니다.
- 들여쓰기는 4칸을 사용합니다.
- 실제 자동 포맷팅 규칙은 프로젝트 설정 파일이 있으면 해당 설정을 우선합니다.

## 15. 추후 보완 예정 항목

아직 합의되지 않은 코드 컨벤션 주제를 모아두는 공간입니다.
