# 17 배수 시설 상세 화면 실시간 동기화 결과

## 작업 결과

- 전역 store에 시설별 마지막 `DRAIN_STATUS_UPDATED` 이벤트를 보관하도록 추가했다.
- 전역 WebSocket 처리 컴포넌트가 기존 대시보드 목록 캐시 갱신과 함께 상태 이벤트를 store에도 전달하도록 연결했다.
- 상세 페이지는 현재 열어 둔 시설의 상태 이벤트만 받아 `detailData`에 즉시 병합한다.
  - 상태, 막힘률, 수위, 유속, 최종 판단, 최근 시각
  - 센서 차트의 최신 포인트
  - 최신 XGBoost 결과와 위험 이력
- 기존 YOLO/XGBoost 개별 이벤트 병합은 유지했다.
- WebSocket이 재연결되면 상세 데이터를 백그라운드에서 보정 조회한다. 기존 화면을 비우지 않으며, 오래된 요청 응답이 새 이벤트 상태를 덮어쓰지 않도록 요청 순서를 확인한다.

## 남은 리스크

- `DRAIN_STATUS_UPDATED`에 없는 CCTV 이미지 URL·YOLO 신뢰도는 마지막 정상 값으로 유지된다.
- AI 기반 통합 테스트에서 해당 세부값까지 즉시 갱신해야 한다는 요구가 확인되면, 백엔드 이벤트 payload 보완을 별도 작업으로 결정한다.
- build 실패는 의존성 설치 상태 문제이므로, 정상 의존성이 갖춰진 환경에서 다시 실행해야 한다.

## 사용자 테스트 방법

### 사용할 데이터와 입력 가이드

AI 분석은 해당 시설의 **가장 최근 센서 데이터**를 사용한다. 따라서 각 시나리오는 반드시 `센서 저장 → AI 분석 시작 → 이벤트 수신 확인` 순서로 실행한다.

#### 1. 시작 전 확인

- 개발 환경에서 backend, ai-service, frontend가 모두 실행 중이어야 한다.
- AI 분석 기능이 활성화되어 있어야 한다. Compose 기준 환경변수는 `COMPOSE_AI_SERVER_ENABLED=true`이다.
- 먼저 대시보드에서 `DR-004` 상세 페이지를 열어 둔다. 화면의 WebSocket 연결 상태가 연결됨이어야 한다.
- 아래 예시는 seed 데이터가 초기 상태인 경우를 기준으로 한다. 이 경우 `DR-004`의 DB 내부 숫자 ID는 일반적으로 `4`다. 데이터가 초기화·추가된 환경에서는 실제 `drains.id`를 확인한 뒤 아래 `4`를 해당 숫자로 바꾼다.

#### 2. 입력 필드 기준

| API | 필드 | 값 기준 |
| --- | --- | --- |
| `POST /api/sensor-data` | `drainId` | DB 내부 숫자 ID. 예시 대상은 `4` |
| `POST /api/sensor-data` | `waterLevelCm` | 수위(cm). 차트에는 m 단위로 변환되어 표시됨 |
| `POST /api/sensor-data` | `flowVelocityMps` | 유속(m/s). 화면의 유량 표시에 사용됨 |
| `POST /api/sensor-data` | `measuredAt` | 이전 데이터보다 늦은 KST ISO-8601 시각. 최신 센서값으로 선택되게 설정 |
| `POST /api/analysis/async-run` | `drainId` | 화면 시설 코드 문자열. 예시 대상은 `"DR-004"` |

#### 3. 시나리오별 입력값

아래 순서대로 실행하면 차트에 시간 순서대로 세 개의 새 포인트가 쌓인다. 실제 최종 위험 상태는 AI 모델의 YOLO/XGBoost 결과에 따라 달라질 수 있으므로, 특정 상태값 자체보다 **입력한 수치와 분석 결과가 상세 화면에 갱신되는지**를 확인한다.

| 순서 | 시나리오 | `waterLevelCm` | `flowVelocityMps` | `measuredAt` | 관찰할 변화 |
| --- | --- | ---: | ---: | --- | --- |
| 1 | 정상 후보 | 15 | 1.2 | `2026-06-22T15:00:00+09:00` | 낮은 수위·상대적으로 높은 유속 포인트, 새 분석 시각 |
| 2 | 주의 후보 | 55 | 0.4 | `2026-06-22T15:10:00+09:00` | 수위 상승·유속 감소 포인트, 위험 이력 추가 |
| 3 | 위험 후보 | 85 | 0.05 | `2026-06-22T15:20:00+09:00` | 높은 수위·낮은 유속 포인트, 최신 상태/판정 갱신 |

#### 4. 각 시나리오 실행 방법

먼저 해당 시나리오의 값으로 센서 데이터를 저장한다. 아래 예시는 3번 위험 후보 데이터다.

```json
POST /api/sensor-data

{
  "drainId": 4,
  "waterLevelCm": 85,
  "flowVelocityMps": 0.05,
  "measuredAt": "2026-06-22T15:20:00+09:00"
}
```

응답이 `success: true`인지 확인한 뒤, 같은 시설의 비동기 AI 분석을 시작한다.

```json
POST /api/analysis/async-run

{
  "drainId": "DR-004"
}
```

분석 시작 응답의 `data.status`가 처리 중 상태인지 확인한다. AI 분석과 callback이 끝날 때까지 잠시 기다린 뒤 상세 페이지를 새로고침하지 않고 화면을 확인한다.

#### 5. 시나리오마다 확인할 화면 항목

| 화면 영역 | 확인 기준 |
| --- | --- |
| 현재 분석 요약 | 수위가 `0.85 m`처럼 cm→m 변환되어 표시되고, 유속·막힘률·위험 점수·판정이 최신 분석 결과로 바뀜 |
| 센서 데이터 추세 | 새 시각의 수위·유속 포인트가 추가되고, 현재 수위/유량 요약이 최신 포인트를 가리킴 |
| AI 모델 판단 정보 | YOLO/XGBoost 이벤트가 제공되면 결과 ID, 분석 상태, 위험 점수, 판단 시각과 이력이 추가됨 |
| 시설 정보 및 현재 상태 | 상태 배지, 막힘 정도, 수위, 유량, 최근 업데이트가 메인 화면과 같은 최신 값으로 갱신됨 |
| 과거 위험 이력 | 최신 분석 시각의 상태·점수가 한 건 추가되며 같은 이벤트가 중복으로 쌓이지 않음 |

1. Docker 개발 환경 또는 AI 분석이 연결된 개발 환경을 실행한다.
2. 브라우저에서 대시보드를 열고, 이벤트를 받을 배수 시설의 상세 페이지를 별도 탭 또는 같은 탭에서 연다.
3. AI 분석 요청을 실행해 해당 시설에 대한 `YOLO_RESULT_UPDATED`, `XGBOOST_RESULT_UPDATED`, `DRAIN_STATUS_UPDATED` 이벤트가 순서대로 발생하도록 한다.
4. 상세 페이지를 새로고침하지 않고 아래 항목이 갱신되는지 확인한다.
   - 상태 배지와 최종 판단 문구
   - 수위, 유속, 막힘률, 위험 점수
   - 센서 차트의 최신 포인트와 센서 요약 값
   - XGBoost 분석 결과와 위험 이력
   - YOLO 이벤트가 제공되는 환경에서는 CCTV/YOLO 분석 카드와 분석 이력
5. 다른 시설의 분석을 실행했을 때 현재 열린 상세 페이지가 바뀌지 않는지 확인한다.
6. 실행 중인 WebSocket 연결을 잠시 끊었다가 복구한 뒤, 상세 페이지가 유지된 상태에서 최신 데이터로 보정되는지 확인한다.

테스트 중 YOLO 이미지 URL 또는 신뢰도만 갱신되지 않는다면, 해당 값이 현재 수신한 이벤트 payload에 포함되는지 먼저 확인한다. 이 작업은 제공된 이벤트를 프론트에 반영하는 범위이며, 이벤트 payload 확장은 후속 백엔드 작업으로 분리한다.
