# 32 상세 실시간 갱신 렌더링 최적화 결과

## 작업 목표

상세 화면에서 WebSocket 이벤트가 수신될 때 상세 데이터 객체가 새로 만들어져, 변경과 무관한 지도·CCTV 표시 영역까지 다시 렌더링될 수 있는 흐름을 줄였다. 실제로 무거운 외부 지도 SDK와 이미지 목록에 한정해 비교 기준을 적용했다.

## 변경 내용

| 대상 | 적용 방식 | 다시 렌더링되는 조건 |
| --- | --- | --- |
| `LocationMapCard` | `memo`와 custom props 비교 | source, 시설 ID, 도로명, 주소, 위험 상태, 위도, 경도 변경 |
| `CctvSnapshotCard` | `memo`와 스냅샷 배열 내용 비교 | 시설명, 스냅샷 수, ID, 이미지 URL, 촬영 시각, 막힘률, 신뢰도, YOLO 상태 변경 |

## 기대 효과

- 수위·유속·판정 문구처럼 지도 위치와 무관한 상태 갱신에서는 Kakao 지도와 마커를 불필요하게 다시 그리지 않는다.
- 위험 상태·좌표·주소가 실제로 바뀌면 지도는 정상적으로 갱신된다.
- 상태 이벤트가 CCTV 이미지/분석 결과를 바꾸지 않으면 이미지 목록과 preview 상태를 불필요하게 다시 계산하지 않는다.
- YOLO 분석 이벤트로 스냅샷 정보가 바뀌면 CCTV 카드는 정상적으로 갱신된다.

## 적용 원칙

비교 함수는 화면에 실제로 표시하는 props만 비교한다. 전체 `DrainFacility` 객체 참조를 비교하지 않아, WebSocket 갱신 과정에서 새 객체가 생성되더라도 관련 값이 같은 경우 렌더링을 건너뛴다. 지도 위험 상태는 marker 색상에 영향을 주므로 비교 대상에 포함했다.

## 검증 결과

| 명령어 | 결과 | 비고 |
| --- | --- | --- |
| `pnpm.cmd lint` | 통과 | 오류 0건. 기존 `FallbackImage`의 `<img>` 관련 성능 경고 1건 유지 |
| `pnpm.cmd build` | 통과 | Next.js production build 성공. `/` 정적 생성 및 `/drains/[id]` 동적 route 확인 |

## 남은 확인 사항

브라우저에서 React DevTools Profiler를 사용할 수 있는 환경이라면, 수위/유속만 바꾸는 WebSocket 이벤트와 YOLO 이미지 변경 이벤트를 각각 재현해 지도와 CCTV 카드의 render 횟수를 비교한다. 이번 작업 환경에서는 개발 서버 기반 profiler 측정은 수행하지 않았다.
