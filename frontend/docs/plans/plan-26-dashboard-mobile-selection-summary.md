# 26 모바일 선택 시설 인라인 요약 및 XGBoost 정보 격자 개선 계획

## 1. 작업 개요

| 항목 | 내용 |
| --- | --- |
| 작업 브랜치 | `style/mobile-ui-refactoring` (사용자가 생성·체크아웃함) |
| 작업 규모 | 중간 작업 — 모바일 대시보드의 선택 시설 확인 흐름과 상세 XGBoost 탭의 정보 격자를 함께 보완한다. |
| 최종 목표 | 모바일에서 관리자가 목록 항목을 선택한 즉시 그 **항목 바로 아래**에서 핵심 수치와 상세 이동을 확인하고, 상세 화면 XGBoost 탭에서는 빈칸 없는 2열×3행 격자로 판단 정보를 읽을 수 있게 한다. |
| 구현 전제 | 현재 `MobileDrainInlineSummary`는 목록 전체의 다음 영역에 표시된다. 이를 목록의 선택 항목 직후로 옮기는 것이 이번 요청의 핵심이다. |
| 비목표 | 위험도 계산·정렬 기준, API/웹소켓 계약, 지도·차트 라이브러리, 전역 디자인 시스템은 변경하지 않는다. |

## 2. 확인한 현황

현재 대시보드의 데이터·선택 상태는 `app/page.tsx`에서 관리하며, 위험 목록과 지도는 같은 `selectedDrainId`를 통해 선택 상태를 공유한다.

```text
지도 마커 또는 위험 시설 목록 클릭
  -> selectDrain(id)
  -> selectedDrainId 갱신
  -> 선택 시설 데이터 계산
  -> lg 미만: 선택된 목록 항목 바로 다음에 MobileDrainInlineSummary 표시
  -> xl 이상: 우측 DrainDetailPanel 표시
```

모바일·태블릿 구간에서 목록 안의 선택 흐름은 다음과 같이 바꾼다.

```text
[위험도 지도]
      ↓
[위험 시설 목록 항목 1]
[위험 시설 목록 항목 2 (선택)]
      ↓
[선택 시설 요약: 상태 / 막힘 / 수위 / 유속 / 최근 업데이트]
[상세 정보 페이지로 이동]
      ↓
[나머지 위험 시설 목록]
```

긴 목록의 마지막까지 스크롤해야 요약을 볼 필요가 없어지므로, 선택·확인·상세 이동을 한 구간에서 끝낼 수 있다. 목록 자체는 전체 시설을 비교하는 역할을 유지하고, 요약 카드는 선택 결과를 확인하고 상세 화면으로 이어 주는 역할을 맡는다.

## 3. 이번 요청의 권장 방향

**모바일의 세로 흐름에서는 목록을 유지하되, 선택한 시설의 간결한 요약과 상세 이동을 선택 항목 바로 다음에 삽입한다.**

목록 항목을 누르는 즉시 상세 페이지로 이동시키거나, 목록을 선택 카드로 교체하지는 않는다. 관리자가 위험 순위를 계속 비교하면서 선택만 바꿀 수 있어야 하고, 상세 이동은 명시적인 버튼으로 분리하는 편이 관제 업무에서 실수와 맥락 상실을 줄인다.

| 화면 구간 | 권장 배치 | 이유 |
| --- | --- | --- |
| 모바일·세로 배치 (`< 1024px`) | 지도 → 목록의 선택 항목 → 선택 시설 요약 → 상세 이동 → 나머지 목록 | 좁은 폭에서 두 패널을 나란히 두지 않아 텍스트·수치가 눌리는 문제를 피하고, 긴 목록에서도 선택 결과를 즉시 확인한다. |
| 일반 데스크톱 (`>= 1024px`, `< 1280px`) | 지도와 목록을 나란히, 선택 요약은 다음 행 | 지도·목록의 비교를 우선하고, 요약은 충분한 가로 폭을 확보한다. |
| 넓은 데스크톱 (`>= 1280px`) | 지도·목록·상세 패널을 나란히 | 관제 화면에서 목록 비교와 상세 확인을 동시에 수행한다. |

선택 요약에는 현재처럼 아래 정보만 유지한다.

- 시설명과 상태
- 막힘률, 수위, 유속
- 최근 업데이트 시각
- `상세 정보 페이지로 이동` 버튼

CCTV 이미지, 긴 판정 문구, 이력 차트처럼 세로 길이를 크게 만드는 정보는 상세 화면에 둔다. 이 경계가 모바일 요약 카드를 또 하나의 상세 페이지로 커지는 것을 막는다.

## 4. 기존 개선 방향을 지키기 위한 결정 기준

`docs/steps/step-19-dashboard-summary-responsive-layout.md`, `step-25-dashboard-route-component-extraction.md`, `step-27-detail-summary-responsive-layout.md`를 확인했다. 이전 개선의 공통 원칙은 빈칸을 기계적으로 채우거나 모든 정보를 같은 크기로 나열하는 것이 아니라, **관제 우선순위와 화면 폭에 맞춰 정보 밀도를 조절하는 것**이다.

따라서 아래 방향으로 되돌아가지 않는다.

| 피해야 할 방향 | 문제 | 유지할 기준 |
| --- | --- | --- |
| 모바일에서 지도·목록·상세를 다단으로 나란히 배치 | 카드 폭과 텍스트가 과도하게 좁아지고 조작 대상이 불분명해진다. | 모바일은 한 열의 순차 흐름을 유지한다. |
| 목록 클릭 즉시 상세 페이지로 이동 | 여러 위험 시설을 비교·재선택하기 어렵고, 목록 맥락을 잃는다. | 클릭은 선택만 수행하고, 상세 이동은 별도 버튼으로 명시한다. |
| 선택 요약에 상세 화면의 이미지·차트·긴 분석을 모두 복제 | 목록 뒤 카드가 길어져 이번 요청의 세로 UX 문제를 다시 만든다. | 요약에는 핵심 수치와 이동 행동만 둔다. |
| 화면 폭과 무관하게 동일한 카드 크기·그리드를 사용 | 모바일에서 빈칸·답답함, 데스크톱에서 과도한 여백이 생긴다. | 기존 breakpoint별 정보 밀도 규칙을 유지한다. |
| 지도·목록·요약의 선택 상태를 각각 따로 관리 | 실시간 갱신 뒤 서로 다른 시설을 표시할 위험이 있다. | 현재의 단일 `selectedDrainId` 흐름을 유지한다. |
| 목록 전체 하단에 선택 요약을 고정 | 긴 목록일수록 선택 직후의 요약·상세 이동이 멀어진다. | 모바일에서는 선택 항목 직후에만 인라인 요약을 표시한다. |

## 5. 추가 UI 개선 후보와 우선순위

이번 요청의 핵심 동작을 유지한 다음, 아래 순서로 작은 범위의 개선을 권장한다.

| 우선순위 | 개선 후보 | 추천 이유 | 범위 |
| --- | --- | --- | --- |
| 1 | 선택 목록 항목의 선택 상태를 상태색이 아닌 공통 강조 규칙으로 명확히 표시 | 현재 선택된 양호·주의 시설도 위험 색상 테두리로 보일 수 있어 상태 의미와 선택 의미가 섞일 여지가 있다. | 목록 항목 스타일과 접근성 표시 |
| 2 | 모바일 선택 요약의 인라인 삽입 | 긴 목록에서도 선택한 카드 바로 아래에서 요약과 상세 이동을 확인하도록 한다. | 목록 렌더링 구조와 요약 위치 |
| 3 | 360px 이하·긴 시설명·날짜의 줄바꿈 점검 | 실제 주소와 업데이트 시각이 길어질 때 카드 높이가 급격히 커지는 것을 방지한다. | 텍스트 clamp, `min-w-0`, 날짜 표시 규칙 |
| 4 | 로딩·오류·빈 목록에서 요약 영역의 일관성 점검 | 정상 데이터에서만 좋은 흐름이 되지 않도록 상태별 메시지와 재시도 위치를 확인한다. | placeholder와 재시도 UI |
| 5 | 상세 화면으로 돌아왔을 때 선택 맥락 유지 여부 점검 | 상세 확인 후 목록 비교를 이어 가는 관제 흐름을 매끄럽게 만든다. | 상세 화면의 뒤로가기·링크 동작 검토 (별도 승인 후) |

1~4와 아래 XGBoost 격자 개선은 대시보드·상세 화면 내부 UI 범위다. 5는 상세 라우트의 탐색 흐름에 영향을 주므로 별도 결정 후 진행한다.

## 6. XGBoost 탭의 2열×3행 정보 격자

현재 `XgboostAnalysisPanel`은 2열 격자에 다섯 개의 일반 항목과 전체 폭의 `최종 판단 문구`를 섞어 사용한다. 이 때문에 `판단 시각` 오른쪽에 빈칸이 생기고, 판정 문구가 길면 카드의 높이가 예측하기 어렵게 늘어난다.

아래처럼 여섯 정보를 모두 동일한 격자 셀로 배치한다.

```text
[ XGBoost Result ID ] [ 위험 상태      ]
[ 참조 Sensor ID    ] [ 참조 YOLO ID  ]
[ 판단 시각         ] [ 최종 판단 문구]
```

| 표시 규칙 | 권장안 |
| --- | --- |
| 격자 | 화면 폭과 관계없이 기본 2열×3행을 유지한다. |
| 셀 크기 | 각 셀은 `min-w-0`과 같은 최소 폭 규칙을 적용하고, 같은 행의 높이를 고정 또는 제한해 내용 길이가 레이아웃을 밀지 않게 한다. |
| 판단 시각 | 한 줄 말줄임표(`truncate`) 처리하고 전체 값은 `title`로 확인 가능하게 한다. |
| 최종 판단 문구 | 최대 두 줄까지만 표시(`line-clamp-2`)하고 전체 문구는 `title`로 확인 가능하게 한다. |
| 상태 배지 | 위험도 색상 체계를 유지하되 셀 크기를 늘리지 않는 `shrink-0` 배지로 표시한다. |
| 누락 값 | 현재처럼 `-`로 표시해 빈 셀이나 레이아웃 변형을 만들지 않는다. |

이 변경은 표시 레이아웃만 다루며, XGBoost 결과 선택·WebSocket 병합·API DTO의 데이터 흐름은 그대로 유지한다.

## 7. 상세 화면의 대시보드 복귀 UX

현재 상세 화면에는 제목 위에 일반 링크 형태의 `대시보드로 돌아가기`가 있다. 상세 화면이 짧을 때는 충분하지만, CCTV·센서 차트·AI 분석·이력을 확인하며 세로로 길어진 뒤에는 화면 최상단까지 다시 올라가야 한다.

### 대안 비교

| 대안 | 장점 | 한계 | 판단 |
| --- | --- | --- | --- |
| 현재처럼 상단 링크만 유지 | 화면을 가리지 않고 구현·학습 비용이 없다. 데스크톱에서는 마우스·큰 화면으로 복귀가 쉽다. | 모바일의 긴 상세 화면에서 복귀하려면 상당히 위로 스크롤해야 한다. | 모바일 관제 흐름에는 부족하다. |
| 모든 화면에서 고정 하단 `대시보드로` 버튼 | 언제나 즉시 복귀할 수 있고 동작이 단순하다. | 데스크톱에서도 내용을 가리고, 모바일에서는 고정 영역이 계속 화면 높이를 차지한다. 브라우저 뒤로가기와 행동이 중복된다. | 전체 화면 공통 적용은 권장하지 않는다. |
| 모바일에서만 고정 하단 복귀 버튼, 데스크톱은 상단 링크 유지 | 긴 모바일 상세 화면에서도 한 번의 탭으로 대시보드로 돌아갈 수 있고, 데스크톱의 정보 밀도를 해치지 않는다. | 안전 영역·마지막 콘텐츠 가림·키보드 노출을 고려해야 한다. | **권장** |
| 스크롤을 내린 뒤에만 나타나는 플로팅 복귀 버튼 | 상단에서는 불필요한 고정을 줄이고, 긴 화면에서만 빠른 복귀를 제공한다. | 스크롤 감지·전환 시점·애니메이션이 추가되어 행동을 예측하기 어렵게 만들 수 있다. | 복귀 버튼과 역할이 달라, 아래의 `위로 가기` 보조 행동으로 제한해 적용한다. |

### 권장 구현 방향

**기존 상단 링크는 유지하고, `< 1024px`의 상세 화면에서 충분히 아래로 스크롤한 경우에만 작은 플로팅 빠른 행동을 제공한다.**

- 전체 폭 고정 바는 콘텐츠와 CCTV·차트 조작 영역을 가리므로 사용하지 않는다.
- 빠른 행동은 `480px` 이상 스크롤한 경우에만 우측 하단에 표시한다. `대시보드` 칩과 `위로 가기` 원형 버튼을 세로로 배치한다.
- 마지막 콘텐츠에는 안전 영역과 플로팅 버튼 영역을 고려한 아래 여백을 추가해 이력·오류 안내·재시도 버튼이 가려지지 않게 한다.
- `대시보드` 칩은 브라우저의 임의의 이전 페이지가 아니라 항상 `/` 대시보드로 이동한다. 직접 링크로 상세에 들어온 경우에도 예측 가능한 복귀 지점을 제공하기 위함이다.
- 같은 세션에서 대시보드 목록에서 상세로 이동했다면 Zustand의 선택 ID는 유지되므로, 해당 시설의 선택 상태를 다시 보여 줄 수 있다. 새로고침·직접 진입 후에는 현재 규칙대로 위험도순 첫 시설이 선택된다.
- 상세 정보가 짧은 경우에도 버튼 위치를 바꾸지 않아, 사용자가 화면마다 다른 복귀 위치를 학습하지 않게 한다.

고정 버튼은 모바일의 시스템 뒤로가기 또는 브라우저 뒤로가기를 대체하지 않는다. 시스템 뒤로가기는 방문 이력을 따르고, 화면의 `대시보드로 돌아가기`는 관제 작업의 안전한 기준 화면으로 복귀하는 명시적 행동이다.

### 모바일 `위로 가기` 보조 버튼

상세 화면의 긴 이미지·차트·이력을 본 뒤 지도와 시설 헤더를 다시 확인할 수 있도록, 모바일에만 `위로 가기` 보조 버튼을 추가한다. 이 버튼은 대시보드로 나가지 않고 현재 상세 화면의 최상단으로 부드럽게 이동한다.

| 항목 | 적용 기준 |
| --- | --- |
| 표시 시점 | 화면을 약 `480px` 이상 아래로 스크롤한 뒤에만 표시한다. 상세 상단에서는 숨겨 불필요한 조작 대상을 늘리지 않는다. |
| 위치 | 모바일 화면 우측 하단에서 `대시보드` 플로팅 칩의 **바로 위**에 둔다. 버튼끼리 최소 `12px` 이상 간격을 확보한다. |
| 크기·터치 | 아이콘만 쓰더라도 최소 `44×44px` 터치 영역을 보장하고, `aria-label="상단으로 이동"`을 제공한다. |
| 시각적 우선순위 | 대시보드 복귀보다 낮은 보조 동작으로, 작은 원형 또는 둥근 사각 버튼과 중립 색상을 사용한다. 위험도 색상은 사용하지 않는다. |
| 동작 | 탭하면 `window.scrollTo({ top: 0, behavior: "smooth" })`로 이동한다. 페이지 이동·선택 시설 상태 변경은 하지 않는다. |
| 충돌 방지 | 안전 영역을 반영하고 마지막 콘텐츠의 하단 여백을 확보한다. 버튼 노출은 모바일에만 제한하며, 스크롤 중에는 조작하지 않도록 단순 표시/숨김 전환만 사용한다. |

이 보조 버튼은 긴 화면에서 지도를 다시 보기 위한 빠른 이동 수단이다. 상시 고정하지 않으므로 스크롤 시작·목록 비교 중 오터치 가능성을 낮추고, 필요가 없으면 나중에 독립적으로 제거할 수 있다.

## 9. 사용자 확인이 필요한 사항

| 확인 항목 | 권장안 | 확인이 필요한 이유 |
| --- | --- | --- |
| 모바일 선택 후 이동 방식 | **현재처럼 목록 선택은 요약 갱신만 하고, 별도 버튼으로 상세 이동** | 비교 중 실수로 화면을 이탈하지 않으며, 사용자가 의도적으로 상세를 연다. |
| 선택 결과 노출 방식 | **선택한 목록 항목 바로 아래의 인라인 요약 카드** | 긴 목록의 끝까지 가지 않고, 선택 직후 핵심 수치와 상세 이동을 확인한다. |
| 선택 카드의 추가 콘텐츠 | **핵심 수치·업데이트 시각·이동 버튼까지만 유지** | 모바일 세로 길이를 다시 늘리지 않는다. |
| 긴 목록에서의 자동 스크롤 | **자동 스크롤 없음** | 선택 요약이 바로 아래에 삽입되므로 추가 화면 이동이 필요 없고, 목록 비교 맥락도 유지된다. |
| 미선택 상태 | **현재처럼 첫 번째 위험도순 시설을 기본 선택** | 데이터가 있는 경우 항상 요약과 상세 이동을 제공해 관제 화면의 첫 판단을 빠르게 한다. 목록이 비었거나 오류이면 요약을 표시하지 않는다. |
| XGBoost 긴 문구 처리 | **2열×3행 고정 격자, 판단 문구는 최대 두 줄 말줄임** | 빈칸과 셀 높이 확장을 없애며, 전체 내용은 hover/focus의 `title`로 확인할 수 있다. |
| 상세 화면 복귀 위치 | **상단 링크 유지, 모바일은 480px 이후 작은 `대시보드` 플로팅 칩 표시** | 긴 상세에서 빠르게 관제 기준 화면으로 복귀하면서, 전체 폭 고정 바로 콘텐츠를 가리는 문제를 없앤다. |
| 복귀 대상 | **항상 대시보드(`/`)** | 직접 링크·새로고침 등 방문 이력과 무관하게 예측 가능한 복귀 지점을 보장한다. |
| 위로 가기 버튼 | **모바일에서 480px 이상 스크롤한 뒤, `대시보드` 플로팅 칩 위에만 표시** | 긴 상세에서 지도·헤더를 빠르게 다시 보되, 상시 노출·오터치·콘텐츠 가림을 줄인다. |
| 선택 강조 색상 | **시설 위험도 색상과 독립된 중립 outline/아이콘 등으로 분리 검토** | `위험`이 아닌 항목도 선택될 수 있으므로 상태 의미를 보존한다. |
| 적용 범위 | **이번 단계는 대시보드 UI와 문서에 한정** | 라우팅·API·전역 상태 변경 없이 요청한 UX를 안전하게 다룬다. |

## 10. 검증 계획

| 검증 | 기대 결과 |
| --- | --- |
| `pnpm lint` | ESLint 오류 없이 통과한다. |
| `pnpm build` | Next.js production build가 통과한다. |
| 360×740 / 375×667 / 390×844 | 목록 선택 뒤 **선택한 항목 바로 다음**에 선택 시설 요약과 상세 이동 버튼이 표시되고, 이후 목록이 이어진다. 시설명·수치·날짜가 넘치지 않는다. |
| 768px | 지도·목록·요약이 한 열 흐름으로 읽히며, 목록 선택과 요약 데이터가 일치한다. |
| 1024px / 1280px 이상 | 현재 breakpoint에 맞춰 요약 또는 상세 패널이 표시되고 지도·목록·선택 시설이 같은 대상을 가리킨다. |
| 지도 선택 및 목록 선택 | 어느 진입점에서 선택해도 같은 요약·상세 링크가 갱신된다. |
| 위험·주의·양호·판단불가 | 상태 색상 체계가 유지되고, 선택 강조가 상태 자체를 오해하게 만들지 않는다. |
| XGBoost 탭 360px / 640px / 1024px | 여섯 정보가 2열×3행으로 빈칸 없이 표시되고, 긴 날짜·판단 문구가 셀 밖으로 넘치거나 행 높이를 과도하게 늘리지 않는다. |
| 상세 화면 360px / 390px | 긴 상세 콘텐츠를 최하단까지 스크롤해도 전체 폭 바 없이 작은 빠른 행동만 보이고, 마지막 콘텐츠·버튼이 안전 영역에 가려지지 않는다. |
| 위로 가기 버튼 | 480px 미만 스크롤에서는 빠른 행동이 보이지 않고, 기준을 넘으면 `대시보드` 칩과 겹치지 않는 위치에 나타난다. 탭하면 현재 상세 화면 최상단의 지도·헤더까지 부드럽게 이동한다. |
| 상세 화면 1024px 이상 | 고정 복귀 버튼은 표시되지 않고 기존 상단 링크만 유지된다. |
| 직접 링크 및 목록에서 상세 진입 | 어느 진입 경로에서도 복귀 버튼은 `/`로 정상 이동하며, 같은 세션의 목록 진입은 선택 시설 상태를 유지한다. |
| 로딩·오류·빈 목록·WebSocket 갱신 | 기존 placeholder, 재시도, 선택 보정, 실시간 갱신 흐름이 회귀하지 않는다. |
