# 26 대시보드 모바일 선택 시설 요약 유지 및 UI 개선 계획

## 1. 작업 개요

| 항목 | 내용 |
| --- | --- |
| 작업 브랜치 | `style/mobile-ui-refactoring` (사용자가 생성·체크아웃함) |
| 작업 규모 | 중간 작업 — 대시보드의 반응형 배치와 선택 시설 확인 흐름을 점검·보완한다. |
| 최종 목표 | 화면 폭이 좁아 한 열로 세로 배치되는 경우, 관리자가 위험 시설 목록에서 항목을 선택한 뒤 목록 아래에서 선택 시설의 핵심 수치와 상세 화면 이동을 바로 확인할 수 있게 한다. |
| 구현 전제 | 현재 코드에는 `MobileDrainInlineSummary`가 있으며, `lg` 미만에서 위험 시설 목록 다음에 선택 시설 요약을 표시한다. 이 동작을 기준 동작으로 유지한다. |
| 비목표 | 위험도 계산·정렬 기준, API/웹소켓 계약, 상세 라우트, 지도·차트 라이브러리, 전역 디자인 시스템은 변경하지 않는다. |

## 2. 확인한 현황

현재 대시보드의 데이터·선택 상태는 `app/page.tsx`에서 관리하며, 위험 목록과 지도는 같은 `selectedDrainId`를 통해 선택 상태를 공유한다.

```text
지도 마커 또는 위험 시설 목록 클릭
  -> selectDrain(id)
  -> selectedDrainId 갱신
  -> 선택 시설 데이터 계산
  -> lg 미만: 목록 아래 MobileDrainInlineSummary 표시
  -> xl 이상: 우측 DrainDetailPanel 표시
```

모바일·태블릿 구간에서 `DashboardMainContent`의 한 열 배치는 다음 순서를 사용한다.

```text
[위험도 지도]
      ↓
[위험 시설 목록]
      ↓
[선택 시설 요약: 상태 / 막힘 / 수위 / 유속 / 최근 업데이트]
      ↓
[상세 정보 페이지로 이동]
```

이는 목록을 누른 뒤 상세 정보를 보기 위해 화면 상단의 지도 또는 별도 패널을 다시 찾지 않아도 되는 흐름이다. 목록 자체는 전체 시설을 비교하는 역할을 유지하고, 요약 카드는 선택 결과를 확인하고 상세 화면으로 이어 주는 역할을 맡는다.

## 3. 이번 요청의 권장 방향

**모바일의 세로 흐름에서는 현재처럼 목록을 먼저 유지하고, 선택한 시설의 간결한 요약과 상세 이동을 목록 바로 다음에 제공한다.**

목록 항목을 누르는 즉시 상세 페이지로 이동시키거나, 목록을 선택 카드로 교체하지는 않는다. 관리자가 위험 순위를 계속 비교하면서 선택만 바꿀 수 있어야 하고, 상세 이동은 명시적인 버튼으로 분리하는 편이 관제 업무에서 실수와 맥락 상실을 줄인다.

| 화면 구간 | 권장 배치 | 이유 |
| --- | --- | --- |
| 모바일·세로 배치 (`< 1024px`) | 지도 → 목록 → 선택 시설 요약 → 상세 이동 | 좁은 폭에서 두 패널을 나란히 두지 않아 텍스트·수치가 눌리는 문제를 피하고, 선택 결과를 목록 직후에 확인한다. |
| 일반 데스크톱 (`>= 1024px`, `< 1280px`) | 지도와 목록을 나란히, 선택 요약은 다음 행 | 지도·목록의 비교를 우선하고, 요약은 충분한 가로 폭을 확보한다. |
| 넓은 데스크톱 (`>= 1280px`) | 지도·목록·상세 패널을 나란히 | 관제 화면에서 목록 비교와 상세 확인을 동시에 수행한다. |

선택 요약에는 현재처럼 아래 정보만 유지한다.

- 시설명과 상태
- 막힘률, 수위, 유속
- 최근 업데이트 시각
- `상세 정보 페이지로 이동` 버튼

CCTV 이미지, 긴 판정 문구, 이력 차트처럼 세로 길이를 크게 만드는 정보는 상세 화면에 둔다. 이 경계가 모바일 요약 카드를 또 하나의 상세 페이지로 커지는 것을 막는다.

## 4. 기존 개선 방향을 지키기 위한 결정 기준

`docs/steps/step-19-dashboard-summary-responsive-layout.md`, `step-25-dashboard-route-component-extraction.md`, `step-27-detail-summary-responsive-layout.md`를 확인했다. 이전 개선의 공통 원칙은 빈칸을 기계적으로 채우거나 모든 정보를 같은 크기로 나열하는 것이 아니라, **관제 우선순위와 화면 폭에 맞춰 정보 밀도를 조절하는 것**이다.

따라서 아래 방향으로 되돌아가지 않는다.

| 피해야 할 방향 | 문제 | 유지할 기준 |
| --- | --- | --- |
| 모바일에서 지도·목록·상세를 다단으로 나란히 배치 | 카드 폭과 텍스트가 과도하게 좁아지고 조작 대상이 불분명해진다. | 모바일은 한 열의 순차 흐름을 유지한다. |
| 목록 클릭 즉시 상세 페이지로 이동 | 여러 위험 시설을 비교·재선택하기 어렵고, 목록 맥락을 잃는다. | 클릭은 선택만 수행하고, 상세 이동은 별도 버튼으로 명시한다. |
| 선택 요약에 상세 화면의 이미지·차트·긴 분석을 모두 복제 | 목록 뒤 카드가 길어져 이번 요청의 세로 UX 문제를 다시 만든다. | 요약에는 핵심 수치와 이동 행동만 둔다. |
| 화면 폭과 무관하게 동일한 카드 크기·그리드를 사용 | 모바일에서 빈칸·답답함, 데스크톱에서 과도한 여백이 생긴다. | 기존 breakpoint별 정보 밀도 규칙을 유지한다. |
| 지도·목록·요약의 선택 상태를 각각 따로 관리 | 실시간 갱신 뒤 서로 다른 시설을 표시할 위험이 있다. | 현재의 단일 `selectedDrainId` 흐름을 유지한다. |

## 5. 추가 UI 개선 후보와 우선순위

이번 요청의 핵심 동작을 유지한 다음, 아래 순서로 작은 범위의 개선을 권장한다.

| 우선순위 | 개선 후보 | 추천 이유 | 범위 |
| --- | --- | --- | --- |
| 1 | 선택 목록 항목의 선택 상태를 상태색이 아닌 공통 강조 규칙으로 명확히 표시 | 현재 선택된 양호·주의 시설도 위험 색상 테두리로 보일 수 있어 상태 의미와 선택 의미가 섞일 여지가 있다. | 목록 항목 스타일과 접근성 표시 |
| 2 | 모바일 선택 요약으로의 이동 피드백 점검 | 긴 목록에서 항목을 누른 뒤 요약 카드가 화면 아래에 있으면 선택 결과를 즉시 알아차리기 어렵다. 자동 스크롤 여부보다 먼저 실제 기기에서 카드 위치와 선택 강조의 충분성을 검증한다. | 수동 UX 검증, 필요 시 제한된 안내 처리 |
| 3 | 360px 이하·긴 시설명·날짜의 줄바꿈 점검 | 실제 주소와 업데이트 시각이 길어질 때 카드 높이가 급격히 커지는 것을 방지한다. | 텍스트 clamp, `min-w-0`, 날짜 표시 규칙 |
| 4 | 로딩·오류·빈 목록에서 요약 영역의 일관성 점검 | 정상 데이터에서만 좋은 흐름이 되지 않도록 상태별 메시지와 재시도 위치를 확인한다. | placeholder와 재시도 UI |
| 5 | 상세 화면으로 돌아왔을 때 선택 맥락 유지 여부 점검 | 상세 확인 후 목록 비교를 이어 가는 관제 흐름을 매끄럽게 만든다. | 상세 화면의 뒤로가기·링크 동작 검토 (별도 승인 후) |

1~4는 대시보드 내부 UI 개선 범위다. 5는 상세 라우트의 탐색 흐름에 영향을 주므로 별도 결정 후 진행한다.

## 6. 예상 변경 범위

사용자 승인 후 실제 개선이 필요할 때에만 아래 범위에서 변경한다.

| 경로 | 목적 |
| --- | --- |
| `components/dashboard/dashboard-main-content.tsx` | breakpoint별 지도·목록·선택 요약 순서와 표시 조건 유지·보완 |
| `components/dashboard/mobile-drain-inline-summary.tsx` | 선택 시설의 핵심 정보와 상세 이동 행동 가독성 개선 |
| `components/drain-risk-list.tsx` | 선택 상태와 위험도 상태를 구분하는 목록 항목 표현 개선 |
| `frontend/docs/steps/step-39-dashboard-mobile-selection-summary.md` | 구현 완료 후 변경 전/후, 검증, 남은 리스크 기록 |

`app/page.tsx`의 query·선택 보정과 Zustand 선택 상태, API DTO, URL 형식은 변경하지 않는다. 새 패키지 설치도 하지 않는다.

## 7. 사용자 확인이 필요한 사항

| 확인 항목 | 권장안 | 확인이 필요한 이유 |
| --- | --- | --- |
| 모바일 선택 후 이동 방식 | **현재처럼 목록 선택은 요약 갱신만 하고, 별도 버튼으로 상세 이동** | 비교 중 실수로 화면을 이탈하지 않으며, 사용자가 의도적으로 상세를 연다. |
| 선택 결과 노출 방식 | **목록 아래의 인라인 요약 카드 유지** | 고정 하단 시트보다 내용을 가리지 않고, 현재 세로 흐름·스크린샷과도 일치한다. |
| 선택 카드의 추가 콘텐츠 | **핵심 수치·업데이트 시각·이동 버튼까지만 유지** | 모바일 세로 길이를 다시 늘리지 않는다. |
| 긴 목록에서의 자동 스크롤 | **기본값은 자동 스크롤 없음; 실제 기기 검증 후 필요할 때만 짧은 안내를 검토** | 갑작스러운 화면 이동은 사용자가 목록을 비교하는 맥락을 잃게 할 수 있다. |
| 선택 강조 색상 | **시설 위험도 색상과 독립된 중립 outline/아이콘 등으로 분리 검토** | `위험`이 아닌 항목도 선택될 수 있으므로 상태 의미를 보존한다. |
| 적용 범위 | **이번 단계는 대시보드 UI와 문서에 한정** | 라우팅·API·전역 상태 변경 없이 요청한 UX를 안전하게 다룬다. |

## 8. 검증 계획

| 검증 | 기대 결과 |
| --- | --- |
| `pnpm lint` | ESLint 오류 없이 통과한다. |
| `pnpm build` | Next.js production build가 통과한다. |
| 360×740 / 375×667 / 390×844 | 목록 선택 뒤 선택 시설 요약과 상세 이동 버튼이 목록 다음에 표시되고, 시설명·수치·날짜가 넘치지 않는다. |
| 768px | 지도·목록·요약이 한 열 흐름으로 읽히며, 목록 선택과 요약 데이터가 일치한다. |
| 1024px / 1280px 이상 | 현재 breakpoint에 맞춰 요약 또는 상세 패널이 표시되고 지도·목록·선택 시설이 같은 대상을 가리킨다. |
| 지도 선택 및 목록 선택 | 어느 진입점에서 선택해도 같은 요약·상세 링크가 갱신된다. |
| 위험·주의·양호·판단불가 | 상태 색상 체계가 유지되고, 선택 강조가 상태 자체를 오해하게 만들지 않는다. |
| 로딩·오류·빈 목록·WebSocket 갱신 | 기존 placeholder, 재시도, 선택 보정, 실시간 갱신 흐름이 회귀하지 않는다. |
