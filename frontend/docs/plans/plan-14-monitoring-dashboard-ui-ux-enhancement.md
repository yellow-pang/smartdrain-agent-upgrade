# 14 모니터링 대시보드 UI/UX 고도화 계획

## 1. 작업 개요

| 항목 | 내용 |

## 2. 현 상태와 문제 정의

현재 대시보드는 `app/page.tsx`에서 지도·위험 목록·상세 패널을 조합한다. `xl` 이상에서는 세 패널을 동시에 표시하지만, 지도·목록·상세 패널이 서로 다른 `max-height`를 사용한다. 목록 항목의 주소·판단 문구가 길어지면 패널마다 스크롤과 콘텐츠 길이가 달라져 관제 화면의 기준선이 흐트러진다.

| 관찰한 불편 | 원인 | 개선 방향 |
| --- | --- | --- |
| 지도·목록·상세 화면의 세로 길이가 맞지 않음 | 각 영역이 독립적인 `max-height`와 콘텐츠 높이를 사용 | 데스크톱 관제 영역에 공통 높이 기준을 두고 각 패널 내부만 스크롤한다. |
| 긴 시설명·주소·판단 문구가 카드 폭과 높이를 밀어냄 | 텍스트 우선순위와 줄 수 제한이 영역별로 다름 | ID·상태·핵심 수치는 고정하고, 설명성 텍스트는 넓이에 따라 1~2줄 clamp 및 전체 내용을 확인할 수 있는 title/접근성 라벨을 제공한다. |
| 라이트 모드에서 하단 스크롤 시 회색·흰색 배경이 섞여 보임 | 페이지 컨테이너·전역 `body`·빈 여백의 배경 책임이 일관되지 않음 | `html`, `body`, Next 루트, 대시보드 셸의 최소 높이와 라이트 배경 토큰을 일치시킨다. |
| 목록의 제목·정렬 영역·카드 테두리가 잘리거나 겹쳐 보임 | 목록 헤더와 스크롤 영역의 경계가 분리되지 않음 | 목록을 고정 헤더 + 독립 스크롤 본문으로 나누고, 상단/하단 여백과 overflow 경계를 명확히 한다. |
| 라이트 모드에서 둥근 카드와 사각 테두리가 겹쳐 보임 | 카드 컨테이너와 내부 항목 모두 강한 border/shadow를 사용 | 외곽 컨테이너는 얇은 테두리와 일관된 반경만 유지하고, 내부 목록은 상태/선택 시에만 대비를 높인다. |
| 메뉴·사용자 아이콘을 눌러도 반응이 없음 | 기능 예정 버튼에 피드백이 없음 | 누른 버튼 기준으로 작고 일시적인 안내 팝오버를 표시하고, 바깥 클릭·Escape·짧은 자동 닫힘을 지원한다. |
| 상세 화면의 위치 지도가 카드 영역을 넘어감 | 상세 카드의 지도 부모는 `220~260px` 높이지만, 공용 `RiskMap`은 중간 화면 이상에서 최소 `420px` 높이를 강제함 | 상세용 지도에는 부모를 넘지 않는 높이 정책을 명시적으로 전달하거나 공용 컴포넌트에 높이 variant를 둔다. 외곽 컨테이너가 임시로 잘라내는 방식은 사용하지 않는다. |
| 다크 모드에서 센서 차트만 흰 카드로 남음 | `SensorTrendChart`와 tooltip·요약 카드가 라이트 전용 색상으로 고정됨 | 차트 컨테이너·그래프·tooltip·요약 카드를 다크 토큰에 맞춘다. 지도 타일과 CCTV 원본 이미지는 외부/원본 콘텐츠로 유지하되 프레임을 테마에 맞춘다. |

## 3. UX 설계 원칙

1. **관제 우선:** 위험 상태, 시설 식별값, 최신 시각, 핵심 수치가 긴 설명보다 먼저 보인다.
2. **한 화면 한 기준선:** 데스크톱에서는 지도·목록·상세의 외곽 높이를 맞추고, 긴 데이터는 페이지 전체가 아니라 해당 패널 안에서 스크롤한다.
3. **상태 색상 일관성:** 양호는 green, 주의는 amber, 위험은 red, 판단불가는 slate 계열을 유지한다. 색상만으로 상태를 판단하지 않도록 레이블도 함께 제공한다.
4. **라이트 모드의 절제된 표면:** 기본 카드에는 과도한 이중 테두리·그림자를 피하고, 선택됨·위험·hover처럼 의미 있는 상태에서만 대비를 올린다.
5. **기능 예정도 응답한다:** 아직 구현하지 않은 메뉴는 무반응 대신 다음 고도화 예정임을 짧게 안내하되, 실제 기능이 있는 것처럼 보이게 하지 않는다.

## 4. 구현 범위와 단계

### 단계 1. 대시보드 셸과 라이트 테마 표면 정리

- `app/page.tsx`의 페이지 루트를 화면 높이를 채우는 flex 구조로 정리하고, 메인 영역과 빈 여백의 라이트 배경을 동일한 토큰으로 통일한다.
- `app/globals.css`에서 `html`, `body`, 앱 루트의 최소 높이와 `background` 적용 기준을 점검한다.
- 대시보드 외곽 카드의 공통 반경·border·shadow 밀도를 조정해 라이트 모드에서 이중 사각 테두리가 도드라지지 않게 한다.
- 다크 모드의 기존 대비와 위험도 색상은 회귀하지 않도록 함께 확인한다.

완료 기준:

- 라이트/다크 모드 모두 화면을 길게 스크롤해도 배경이 갑자기 다른 색으로 전환되지 않는다.
- 가로·세로 스크롤이 의도치 않게 생기지 않는다.

### 단계 2. 데스크톱 3패널 관제 레이아웃 정렬

- `xl`/`2xl` 기준에서 지도·목록·상세 패널이 같은 관제 높이를 사용하도록 CSS grid와 높이 토큰을 정리한다.
- 지도는 패널 높이를 채우고, 위험 목록과 상세 패널은 헤더/핵심 정보는 고정한 채 본문만 `overflow-y-auto`로 처리한다.
- 넓은 화면에서는 상세 패널의 최소 너비를 확보해 주소·수치·상태 배지가 불필요하게 좁아지지 않게 한다. 공간이 부족한 화면은 기존 인라인 요약 레이아웃으로 자연스럽게 전환한다.
- 패널 내부 스크롤바는 현재의 얇은 스타일을 유지하되, 스크롤 컨테이너 경계를 명확히 한다.

완료 기준:

- 1280px, 1440px, 1536px 이상에서 세 패널 외곽의 상·하단 기준선이 일치한다.
- 목록 또는 상세 데이터가 길어도 지도 높이와 페이지 전체 레이아웃이 밀리지 않는다.

### 단계 3. 위험 목록과 상세 정보의 정보 위계 개선

- `components/dashboard/drain-risk-list.tsx`, `drain-risk-list-item.tsx`를 기준으로 목록 헤더와 스크롤 본문을 분리한다.
- 목록 항목은 순위·시설 ID·상태 배지·시설명·핵심 수치·최신 시각의 순서로 재배치한다. 판단 사유는 1~2줄로 제한한다.
- 상세 패널은 상태/판정/최신 시각을 먼저, 막힘·수위·유량을 다음, CCTV·주소 등 보조 정보를 뒤에 둔다.
- 제목·주소·판정 문구에 영역별 `min-w-0`, line-clamp, `break-words`를 적용한다. 잘린 텍스트는 `title` 또는 접근 가능한 설명으로 전체 값을 제공한다.
- 선택·hover·위험 상태를 명확하게 구분하되, 선택되지 않은 카드에는 불필요한 강한 테두리를 적용하지 않는다.

완료 기준:

- 긴 한글 주소와 긴 판단 사유가 카드 폭을 넘어가거나 헤더를 밀어내지 않는다.
- 목록 첫 항목, 마지막 항목, 정렬 선택 영역이 스크롤/둥근 모서리에 의해 잘리지 않는다.
- 상태를 색상, 텍스트 레이블, 아이콘/배지 중 둘 이상으로 식별할 수 있다.

### 단계 4. 메뉴·사용자 버튼의 기능 예정 피드백

- 재사용 가능한 작은 안내 팝오버(예: `components/coming-soon-popover.tsx`)를 추가하거나 기존 UI primitive로 충분하면 해당 구조를 재사용한다.
- 햄버거와 사용자 버튼 클릭 시 각각의 버튼을 기준으로 “메뉴 기능은 준비 중입니다”, “사용자 기능은 준비 중입니다” 같은 짧은 안내를 노출한다.
- 팝오버는 약 2~3초 뒤 자동으로 닫히며, 바깥 클릭·Escape·다른 버튼 클릭에도 닫힌다.
- `aria-expanded`, `aria-describedby` 등 키보드와 보조기술 사용자에게도 현재 상태와 안내 내용을 전달한다.

완료 기준:

- 버튼 클릭마다 해당 버튼 주변에 안내가 표시되고 자동으로 사라진다.
- 모바일의 좁은 화면에서도 팝오버가 화면 밖으로 잘리지 않는다.

### 단계 5. 상세 화면 지도 범위와 센서 차트 테마 정리

- `app/drains/[id]/page.tsx`의 `LocationMapCard`와 공용 `RiskMap`의 최소 높이 규칙을 분리한다.
  - **권장 구현:** `RiskMap`에 dashboard/detail 용도 또는 `className` 기반의 높이 옵션을 제공한다. 상세 지도는 부모가 정한 `220~260px` 영역을 정확히 채우며, 대시보드 지도만 현재의 큰 최소 높이를 유지한다.
  - 지도 SDK는 컨테이너 크기 변경 뒤 올바르게 다시 배치되는지 확인한다.
  - 카드에 `overflow-hidden`만 추가해 넘침을 감추는 방식은 지도 조작 UI·라벨을 자를 수 있으므로 사용하지 않는다.
- `components/sensor-trend-chart.tsx`의 카드·보조 요약 카드·그래프 grid·축 라벨·tooltip을 라이트/다크 테마 모두에서 읽기 쉬운 토큰으로 변경한다.
  - **권장 방향: 다크 모드에서 차트도 다크 표면으로 전환.** 관제 화면 전체가 다크 배경일 때 차트만 흰색이면 시선이 과도하게 쏠리고 페이지의 표면 체계가 끊긴다. 짙은 slate 표면 위에 저대비 grid, 충분히 밝은 축/범례, 현재의 청록·초록·amber·red 선을 사용해 데이터 식별성을 유지한다.
  - CCTV 이미지와 지도 타일은 원본/외부 콘텐츠라 흰색을 유지할 수 있다. 대신 카드 프레임·caption·조작 요소는 다크 테마 토큰을 적용해 주변 UI와 연결한다.
  - `Tooltip`까지 테마화하여 hover 시 흰 팝업이 갑자기 나타나지 않게 한다.

완료 기준:

- 상세 화면 지도는 360px부터 데스크톱까지 카드의 지정 영역을 넘지 않고, 범례·확대/축소·선택 라벨도 정상적으로 보인다.
- 다크 모드의 센서 차트와 요약 수치가 흰 배경 없이도 축·선·임계선·tooltip을 명확히 구분한다.
- 라이트 모드에서는 기존의 흰 차트 표면과 가독성을 유지한다.

### 단계 6. 반응형·테마·접근성 회귀 확인

- 모바일(360px), 태블릿(768px), 노트북(1280px), 데스크톱(1440px/1536px 이상)에서 레이아웃과 텍스트를 확인한다.
- 라이트·다크·시스템 테마 전환 뒤 배경, border, 상태 색상, 스크롤 영역을 확인한다.
- 긴 주소/판단 사유, 목록 다수, 상세 정보 다수의 데이터를 기준으로 overflow를 확인한다.

## 6. 사용자 확인이 필요한 결정

이번 구현은 기존 대시보드의 디자인 언어를 유지하는 **부분 고도화**로 제한한다. 따라서 새 디자인 시스템, 새 아이콘 세트, 지도/차트 라이브러리 교체, 실제 메뉴·사용자 인증 기능은 포함하지 않는다.

| 항목 | 권장안 | 이유 |
| --- | --- | --- |
| 데스크톱 공통 패널 높이 | 화면 높이를 고려한 `clamp()` 기반 높이와 내부 스크롤 | 768px 안팎의 노트북에서도 콘텐츠가 과도하게 길어지지 않는다. |
| 긴 텍스트 정책 | 목록은 설명 2줄, 상세 제목/주소는 영역별 2줄 후 전체값 제공 | 정보 밀도와 식별 가능성을 함께 확보한다. |
| 메뉴/사용자 안내 문구 | 기능 예정 사실만 알리는 짧은 팝오버 | 작동하는 기능처럼 오해하지 않게 한다. |
| 상세 지도 overflow | 공용 지도에 dashboard/detail 높이 variant를 도입 | 상세 카드의 작은 지도와 대시보드 대형 지도를 하나의 최소 높이로 강제하지 않는다. |
| 다크 모드 센서 차트 표면 | **차트도 다크 표면으로 전환 (권장)** | 다크 관제 화면의 시각적 연속성을 지키고 흰 차트의 눈부심을 줄인다. 축·선·tooltip은 별도 대비 조정이 필요하다. |

## 7. 검증 계획

| 검증 | 기대 결과 |
| --- | --- |
| `pnpm lint` | ESLint 오류 없이 통과 |
| `pnpm build` | Next.js production build 통과 |
| 브라우저 수동 확인 | 360px, 768px, 1280px, 1440px 이상에서 텍스트 넘침·패널 잘림 없음 |
| 테마 수동 확인 | 라이트/다크/시스템 전환, 긴 페이지 스크롤의 배경 일관성, 센서 차트·tooltip의 선/축/임계선 대비 유지 |
| 상세 지도 수동 확인 | 카드 범위 내 지도 표시, 지도 조작 UI·범례·선택 라벨 정상 노출, 창 크기 변경 후 지도 배치 정상 |
| 상호작용 수동 확인 | 지도/목록 선택, 헤더 알림, 메뉴·사용자 기능 예정 팝오버의 클릭·Escape·자동 닫힘 정상 |
| 접근성 점검 | 키보드 focus와 버튼 이름, 팝오버 상태가 유지됨 |

## 8. 리스크와 대응

| 리스크 | 대응 |
| --- | --- |
| 고정된 데스크톱 패널 높이가 작은 노트북에서 답답할 수 있음 | `clamp()`와 브레이크포인트를 사용하고 1280px 이하에서는 기존 인라인 상세 레이아웃을 유지한다. |
| 지도 SDK가 부모 높이 변경을 즉시 반영하지 않을 수 있음 | 패널 크기 변경 뒤 지도 relayout 처리 여부를 기존 지도 컴포넌트 기준으로 확인한다. |
| 과도한 line-clamp가 현장 판단 정보를 숨길 수 있음 | 상태·시설 ID·핵심 수치는 clamp하지 않고, 부가 설명만 제한하며 전체값 접근 수단을 남긴다. |
| 안내 팝오버가 헤더 알림 패널과 충돌할 수 있음 | 한 번에 하나의 헤더 보조 패널만 열리도록 상태를 정리하고, z-index·포커스 동작을 수동 확인한다. |
| 지도 높이를 variant로 분리한 뒤 SDK가 이전 크기를 유지할 수 있음 | 지도 컨테이너 크기 변경과 화면 resize 뒤 SDK relayout 동작을 확인하고, 필요 시 지도 전용 relayout 처리를 추가한다. |
| 다크 차트에서 선·임계선 대비가 약해질 수 있음 | 그래프 격자는 낮은 대비로 두되, 데이터 선·임계선·axis text·tooltip은 WCAG 읽기 수준을 확인한다. |

## 9. 승인 후 실행 순서

1. 기존 대시보드 컴포넌트의 높이·overflow·텍스트 처리 위치를 정확히 점검한다.
2. 단계 1~2의 셸·3패널 레이아웃을 먼저 적용하고 라이트/다크 배경을 확인한다.
3. 단계 3의 목록·상세 정보 위계를 적용한다.
4. 단계 4의 기능 예정 팝오버를 추가한다.
5. 단계 5의 상세 지도 높이와 센서 차트 다크 테마를 적용한다.
6. lint/build 및 화면 크기·테마별 수동 검증을 수행한다.
7. 실제 변경·검증 결과·남은 리스크를 `docs/steps/step-16-monitoring-dashboard-ui-ux-enhancement.md`에 기록한다.
