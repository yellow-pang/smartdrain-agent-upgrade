# 25 프로젝트 기준 문서 정합성 점검 및 갱신 계획

## 1. 작업 개요

| 항목 | 내용 |
| --- | --- |
| 작업 브랜치 | `docs/update-project-docs` |
| 작업 규모 | 큰 작업 — 루트 `docs`의 현재 기준 문서와 신규 운영·검증 기준 문서를 갱신한다. |
| 최종 목표 | 현재 코드·migration·배포 설정·테스트를 근거로 문서의 현재 구현, 미검증 사항, 향후 과제를 명확히 구분한다. |
| 수정 범위 | 루트 `docs/**/*.md`와 `frontend/**` 내부 문서 중 근거가 확인된 파일만 최소 수정한다. |
| 제외 범위 | 애플리케이션 코드, 설정, migration, 테스트, 이미지, lockfile 및 루트 `README.md` 수정 |

## 2. 작업 전 확인 결과

| 항목 | 확인 결과 | 처리 방침 |
| --- | --- | --- |
| Git 작업 트리 | 미커밋 변경 없음 | 기존 사용자 변경을 덮어쓸 위험 없이 문서 조사를 시작한다. |
| 현재 브랜치·기준 커밋 | `docs/update-project-docs` · `c19ebb5b` | 결과 문서에 기준으로 기록한다. |
| 원격 비교 | `origin/dev`를 fetch 후 비교 | `origin/dev`에는 현재 브랜치에 없는 대규모 README 갱신이 있다. |
| 루트 README | 다른 작업자의 갱신 가능성이 높음 | 사용자 요청대로 직접 수정하지 않고 `docs/14`에 반영 제안만 기록한다. |
| 이력 문서 | `frontend/docs/plans`, `steps`, `pr`가 다수 존재 | 구현 사실을 덮어쓰지 않으며, 현행 기준은 신규 루트 문서로 안내한다. |

## 3. 근거와 판단 원칙

문서가 서로 다를 때 다음 우선순위로 판단한다.

1. 현재 애플리케이션 코드와 API schema
2. Alembic migration과 ORM model
3. Docker Compose, Nginx, Jenkinsfile 및 배포 스크립트
4. 테스트 코드와 이번 작업에서 실제로 실행한 검증
5. 최신 원격 README
6. 루트 `docs/01`~`docs/13`
7. 서비스별 README, frontend Plan·Step·PR 문서

- 확인하지 못한 모델 버전, 운영 인프라 상태, 외부 서비스 설정은 추측하지 않고 `확인 필요`로 남긴다.
- 현재 구현과 미래 고도화, 자동 검증과 문서상 수동 검증 기록을 분리한다.
- 명백한 불일치만 기존 기준 문서에서 최소 수정하고, 구조·표현의 전면 개편은 하지 않는다.

## 4. 수행 단계

| 단계 | 작업 | 완료 기준 |
| --- | --- | --- |
| 1 | 코드·migration·API·WebSocket·AI callback·배포 설정·테스트 구조를 읽고 사실 목록을 만든다. | 모든 신규/수정 문장에 코드 또는 설정 근거가 있다. |
| 2 | 루트 `docs/01`~`docs/13`과 서비스별·frontend 문서를 분류하고, endpoint·데이터 모델·용어·배포 설명·링크 불일치를 대조한다. | 현재 기준, 이력, 중복, 확인 필요 문서를 구분한다. |
| 3 | `docs/verification/14_구현현황_및_검증결과.md`를 작성한다. | 구현·검증·미검증·정합성 수정 근거·README 반영 제안이 추적 가능하다. |
| 4 | `docs/reference/15_AI_모델_아티팩트_관리_명세.md`, `docs/verification/16_테스트_전략_및_E2E_검증.md`, `docs/verification/17_배포_운영_런북.md`를 작성한다. | 모델·테스트·운영 절차가 실제 저장소 기준이며 미확인 값은 표시된다. |
| 5 | 불일치가 확정된 기존 현재 기준 문서와 명확한 깨진 링크만 최소 수정한다. | 이력 문서의 당시 기록을 훼손하지 않고 현재 문서와 연결된다. |
| 6 | `git diff --check`, 링크·경로 확인 및 실행 가능한 정적 검증을 수행한다. | 변경 파일, 검증 결과, 남은 위험이 문서와 결과 보고에 남는다. |

## 6. 사용자 확인이 필요한 사항

| 확인 항목 | 권장 방향 | 이유와 영향 |
| --- | --- | --- |
| 루트 README 처리 | **수정하지 않고 `docs/14`에 링크 추가 제안만 기록한다.** | `origin/dev`에 대규모 README 변경이 있어 현재 브랜치에서 수정하면 다른 작업의 내용을 덮어쓸 수 있다. 또한 사용자가 README는 나중에 수정한다고 지정했다. |
| Plan·Step·PR 처리 | **본문은 수정하지 않고, 현행성 오해 위험이 큰 문서만 깨끗한 상태에서 짧은 이력 안내를 추가한다.** | 작업 당시의 기록을 보존하면서 현재 기준 문서로 안전하게 유도한다. 대량 상단 안내 추가는 변경량과 충돌 위험을 늘린다. |
| 기존 핵심 문서 수정 기준 | **코드로 확정한 불일치와 명백한 깨진 링크만 최소 수정한다.** | 문서의 원래 구조와 발표 자료 연결을 유지하고, 추정에 따른 과도한 재작성 위험을 줄인다. |
| frontend 문서 변경 추적 | **변경마다 기존 설명, 코드 근거, 변경 이유, 영향 범위를 `docs/14`와 완료 기록에 남긴다.** | 이후 frontend 구현·문서 변경의 이유와 현재 기준을 함께 추적할 수 있다. |
| 실행 검증 범위 | **문서 확인 후 의존성이 이미 준비된 검증만 실행하고, Docker·모델·외부 서비스가 필요한 항목은 `미실행` 사유로 구분한다.** | 환경을 변경하거나 모델·비밀값을 요구하지 않으면서 검증 기록의 신뢰도를 유지한다. |
| 배포 경로 표현 | **저장소에 고정된 현재 경로와 일반화한 권장 경로를 분리해 기록한다.** | 특정 개발자 계정·VM 경로를 운영 표준처럼 오해하는 일을 막는다. |

## 7. 리스크와 대응

| 리스크 | 대응 |
| --- | --- |
| 원격 README 또는 다른 팀원 문서와 충돌 | README는 수정하지 않고, 작업 전후 Git 상태를 재확인한다. |
| 모델 artifact가 저장소에 없어 세부 정보를 확정할 수 없음 | 모델명·학습일·성능·SHA256을 추정하지 않고 확인 필요 목록과 수집 절차를 기록한다. |
| 이력 문서가 현재 상태로 오해될 수 있음 | `docs/14`를 canonical 현재 기준으로 지정하고, 필요한 경우에만 짧은 상단 안내를 제안한다. |
| 테스트 미실행 결과를 통과로 오해 | 실제 실행 결과, 코드 확인, 기존 문서 기록을 각각 다른 상태로 표기한다. |

## 8. 승인 후 첫 작업

코드·설정·migration·테스트에서 API, callback, 데이터 모델, 배포 경로의 사실을 수집한 뒤, 기존 핵심 문서의 불일치 후보를 표로 확정하고 신규 `docs/14`~`docs/17` 초안을 작성한다.
