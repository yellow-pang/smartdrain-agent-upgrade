# 21 프론트엔드 컴포넌트화·성능·안정성 리팩터링 계획

## 1. 작업 개요

| 항목        | 내용                                                                                                                                                            |
| ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 작업 브랜치 | `refactor/frontend-component-performance-security`                                                                                                              |
| 작업 규모   | 큰 작업 — 대시보드·상세 화면·공통 컴포넌트·데이터 경계를 단계적으로 점검한다.                                                                                   |
| 최종 목표   | 화면 동작과 API·WebSocket 계약을 유지하면서, 두꺼운 route 컴포넌트를 역할별로 분리하고 불필요한 렌더링·반복 계산·취약한 입력 처리 가능성을 줄인다.              |
| 작업 범위   | `app/`, `components/`, `lib/`, `store/` 및 `frontend/docs/` 작업 기록                                                                                           |
| 비목표      | 새 패키지 설치, 라우팅 구조 변경, 상태 관리 라이브러리 교체, 지도·차트 라이브러리 교체, 백엔드 API/WebSocket 계약 변경, mock data 대규모 교체, 전면 디자인 변경 |

이번 작업은 한 번에 구조를 바꾸지 않는다. 각 단계는 독립적으로 검증 가능한 작은 변경 묶음으로 끝내며, 문제가 생기면 직전 검증 완료 지점에서 원인을 좁힐 수 있게 한다.

## 2. 현재 확인 결과와 리팩터링 원칙

- `app/page.tsx`는 대시보드 데이터 조합, 선택 시설 동기화, 요약 바, 로딩/오류 상태, 모바일 시설 요약, 레이아웃을 한 파일에서 맡고 있다.
- 대시보드에는 이미 `memo`, `useMemo`, `useCallback`이 일부 적용되어 있다. 따라서 이를 기계적으로 늘리지 않고, 측정 또는 렌더링 흐름상 근거가 있는 곳만 개선한다.
- 지도·목록·상세 화면은 모두 동일한 시설 선택과 위험도 상태를 보여야 한다. 분리 후에도 `good/caution/danger/unknown` 색상과 선택 동작, 빈 상태, 오류 상태를 바꾸지 않는다.
- API 응답, WebSocket 이벤트, 이미지 URL, route parameter는 신뢰할 수 없는 외부 입력으로 본다. 화면 컴포넌트가 이를 직접 가정하지 않도록 기존 adapter·helper 경계를 우선 확인한다.
- 기존 API DTO의 nullable 처리, 수위 `cm`·유속 `m/s`·막힘률 `%` 표기 기준은 유지한다.

## 3. 단계별 작업 및 검증

| 단계                   | 변경 내용                                                                                                                                                                                                                                          | 완료 기준                                                                                 | 단계 종료 검증                                                     |
| ---------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------- | ------------------------------------------------------------------ |
| 0. 기준선 확보         | 실제 파일 책임, 데이터 흐름, 큰 컴포넌트, 현재 lint/build 결과를 기록한다. 위험도 4종·로딩·오류·빈 목록·선택 변경을 회귀 시나리오로 고정한다.                                                                                                      | 이후 변경을 비교할 기준선과 우선순위가 문서화된다.                                        | `pnpm lint`, `pnpm build`, 대시보드·상세 수동 점검                 |
| 1. 대시보드 route 분리 | `app/page.tsx`에서 요약 바, 요약 skeleton/오류 상태, 모바일 선택 시설 요약, 대시보드 레이아웃 조합을 책임별 컴포넌트로 추출한다. 선택 ID 보정과 query 호출의 소유자는 한 곳에 유지한다.                                                            | 지도↔목록 선택, 새로고침, 모바일/데스크톱 상세 표시가 기존과 동일하다.                    | lint/build 및 360px·768px·1280px 수동 점검                         |
| 2. 상세 route 분리     | `app/drains/[id]/page.tsx`의 데이터 준비, 로딩/오류/빈 상태, 시설 요약, 분석·센서·이력 섹션을 역할별로 나눈다. route parameter와 API 응답의 누락·형식 오류 처리를 명확히 한다.                                                                     | 유효/없는 시설 ID, API 실패, 이미지·센서 이력 누락, 모든 위험도 상태가 안전하게 표시된다. | lint/build 및 상세 화면 회귀 시나리오                              |
| 3. 큰 공통 UI 정리     | 실제 중복과 책임 과다를 확인한 뒤 지도, 위험 목록, 상세 패널, 차트 중 우선순위가 높은 컴포넌트만 작은 표시 컴포넌트와 순수 helper로 나눈다. 기존 public props는 특별한 이유 없이는 유지한다.                                                       | 추출한 컴포넌트의 입력·출력이 명확하고, 기능 차이가 없다.                                 | 컴포넌트별 상태 수동 점검, lint/build                              |
| 4. 보안·견고성 점검    | `dangerouslySetInnerHTML`, 외부 URL 렌더링, 링크 생성, route/query parameter 사용, API/WS payload 변환, 오류 메시지 노출을 검색·점검한다. 필요한 경우 allowlist/형식 검증/null fallback을 adapter 또는 helper에 둔다.                              | 외부 입력을 HTML로 직접 주입하지 않고, 유효하지 않은 값은 안전한 오류·빈 상태로 처리한다. | lint/build, 잘못된 ID·누락값·예상 밖 상태값 수동 확인              |
| 5. 성능 개선           | React DevTools Profiler 또는 재현 가능한 렌더링 관찰로 병목을 확인한다. 정렬·검색·파생 데이터는 데이터 변경 시에만 계산하고, props 안정화와 `memo`는 실제로 반복 렌더링되는 무거운 자식에 한정한다. 지도 SDK 및 차트의 불필요한 재생성도 확인한다. | 최적화 전후 근거와 기대 효과가 기록되고, 가독성을 해치지 않는다.                          | lint/build, 선택·재조회·WebSocket 갱신 시 렌더링/기능 점검         |
| 6. 통합 회귀·기록      | 전체 변경을 재검토하고 단계별 검증 결과, 남은 리스크, 후속 제안을 step/PR 문서에 기록한다.                                                                                                                                                         | 기능·보안·성능 변경의 근거와 검증 결과를 추적할 수 있다.                                  | `pnpm lint`, `pnpm build`, `git diff --check`, 핵심 화면 수동 점검 |

## 4. 권장 진행 단위

한 단계는 가능한 한 하나의 논리적 변경 묶음으로 커밋 가능한 상태를 유지한다. 예를 들어 1단계는 대시보드 route 분리만 다루고, 4단계의 입력 검증이나 5단계의 성능 최적화를 섞지 않는다. 각 단계가 끝날 때 다음 순서로 반복한다.

```text
현재 동작 확인
  -> 작은 책임 단위 추출 또는 경계 보강
  -> 정적 검증(lint/build)
  -> 대시보드/상세 핵심 흐름 수동 확인
  -> 변경·검증·리스크 기록
  -> 다음 단계 승인
```

## 5. 예상 파일 범위

| 구분          | 우선 확인 경로                                                                                            | 목적                                                                   |
| ------------- | --------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------- |
| route         | `app/page.tsx`, `app/drains/[id]/page.tsx`                                                                | route는 데이터 연결과 화면 조합 중심으로 얇게 만든다.                  |
| dashboard     | `components/dashboard/`                                                                                   | 지도·위험 목록·상세 패널의 책임과 props 경계를 정리한다.               |
| shared UI     | `components/sensor-trend-chart.tsx`, `components/cctv-snapshot-card.tsx`, `components/fallback-image.tsx` | 차트·이미지·상태 UI의 큰 렌더링 단위와 외부 입력 처리를 점검한다.      |
| data boundary | `lib/api/`, `lib/query/`, `lib/websocket/`, `lib/risk.ts`                                                 | API/WS DTO 변환, 상태값 정규화, 오류 처리를 화면 밖에 둔다.            |
| state         | `store/drain-store.ts`                                                                                    | 선택 상태와 WebSocket 갱신이 필요한 화면만 다시 렌더링하는지 확인한다. |

위 경로는 조사 대상이며, 실제 수정 파일은 각 단계 착수 전에 좁혀 확정한다.

## 6. 사용자 확인이 필요한 사항

| 확인 항목      | 권장 방향                                                   | 이유                                                                                               |
| -------------- | ----------------------------------------------------------- | -------------------------------------------------------------------------------------------------- |
| 시작 단계      | **0단계 기준선 확보 후 1단계 대시보드 route 분리부터 진행** | 현재 가장 두꺼운 `app/page.tsx`를 먼저 작게 만들되, 기능 변경 없이 안전하게 효과를 확인할 수 있다. |
| 단계 승인 방식 | **단계별 검증 결과를 보고한 뒤 다음 단계 진행**             | 여러 화면의 선택·실시간 흐름을 잃지 않고 문제 범위를 좁힐 수 있다.                                 |
| 보안 변경 범위 | **프론트엔드 입력 경계의 방어만 적용**                      | API 인증·권한·서버 검증은 백엔드 계약 변경이므로 이번 브랜치 범위에서 제외한다.                    |
| 성능 판단 기준 | **프로파일링 또는 반복 렌더링 근거가 있을 때만 최적화**     | `memo`와 hook을 과도하게 넣어 복잡도만 늘리는 일을 막는다.                                         |
| 시각적 변경    | **기존 디자인과 UX를 유지**                                 | 리팩터링 효과와 디자인 변경 효과를 분리해 회귀를 쉽게 확인한다.                                    |

## 7. 단계 공통 회귀 시나리오

1. 대시보드에서 지도 또는 위험 목록을 선택하면 같은 시설이 강조되고 상세 정보가 바뀐다.
2. 선택 시설이 사라지거나 목록이 갱신되면 유효한 첫 시설 또는 빈 상태로 안전하게 전환된다.
3. 양호·주의·위험·판단불가 상태가 지도·목록·상세에서 같은 의미와 색상으로 표시된다.
4. API 로딩, 실패, 빈 배열, nullable 센서/이미지 데이터에서 화면이 깨지지 않는다.
5. WebSocket 상태 변경 후 지도·목록·상세의 표시가 일관되게 갱신된다.
6. 모바일 360px, 태블릿 768px, 데스크톱 1280px에서 텍스트 겹침과 가로 넘침이 없다.

## 8. 리스크와 대응

| 리스크                                                 | 대응                                                                                                           |
| ------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------- |
| 컴포넌트 추출 과정에서 선택 상태의 소유자가 분산됨     | 선택 ID 보정과 상태 변경 함수를 route/container 한 곳에서 관리하고, 표시 컴포넌트에는 필요한 props만 전달한다. |
| 성능 최적화가 stale props 또는 WebSocket 미갱신을 유발 | `memo` 적용 전후에 선택·재조회·실시간 갱신을 모두 확인하고, 비교 함수는 특별한 근거가 없으면 추가하지 않는다.  |
| 보안 점검이 API 계약 변경으로 확장됨                   | 프론트 adapter의 방어적 처리까지만 적용하고 계약 변경이 필요하면 별도 승인 요청으로 분리한다.                  |
| 큰 컴포넌트를 한 번에 재구성해 회귀 원인을 찾기 어려움 | 단계별로 작게 변경하고 lint/build/수동 확인을 통과한 상태만 다음 단계의 기준으로 삼는다.                       |
| 코드 정리와 UI 리디자인이 섞임                         | 레이아웃·문구·색상은 의도적으로 유지하고, 사용자 요청이 있을 때만 별도 디자인 작업으로 분리한다.               |

## 9. 승인 후 첫 실행 항목

사용자가 권장 방향을 승인하면 0단계에서 실제 기준선 결과와 대시보드 데이터·선택 상태 흐름을 기록한 뒤, 1단계에서 `app/page.tsx`만 대상으로 컴포넌트 분리를 진행한다. 각 단계 완료 후에는 별도 step 문서에 실제 변경 파일, 검증 결과, 남은 리스크를 기록한다.
