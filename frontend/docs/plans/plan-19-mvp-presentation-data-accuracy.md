# 19 MVP 발표용 데이터 정확성 개선 계획

## 1. 작업 개요

| 항목 | 내용 |
| --- | --- |
| 권장 브랜치 | `feat/frontend-mvp-feedback` |
| 작업 규모 | 중간 작업 — 대시보드·상세 화면·API adapter·차트에 걸친 표시 의미와 최신 데이터 선택을 바로잡는다. |
| 최종 목표 | 발표 화면에서 API 계약과 같은 단위·시간순서·위험도 의미를 표시하고, 실제로 지원하지 않는 기능을 제공하는 것처럼 보이지 않게 한다. |
| 작업 범위 | `frontend/app`, `frontend/components`, `frontend/lib` 및 작업 문서 |
| 비목표 | 백엔드 API·WebSocket 계약 변경, 새 패키지 도입, 전역 상태 구조 개편, 기간별 조회 API 구현, 전체 디자인 리뉴얼 |

## 2. 확인한 기준과 현재 문제

프로젝트 MVP와 통합 API 명세는 수위를 `waterLevelCm`(cm), 유속을 `flowVelocityMps`(m/s), 막힘률을 `obstructionRatio`(0~1 ratio, 화면에서 %)로 정의한다. 관리자는 지도와 상세 화면에서 최종 위험도(`riskLevel`)와 그 근거가 되는 최신 센서·이미지 분석 데이터를 일관되게 확인해야 한다.

현재 코드에는 다음과 같은 발표 리스크가 있다.

| 우선순위 | 문제 | 영향 |
| --- | --- | --- |
| 필수 | 수위를 `waterLevelPct`로 변환해 표시 | 시설별 기준 높이 없이 cm를 %처럼 보여 의미가 달라진다. |
| 필수 | 센서 이력이 최신순일 때 마지막 항목을 현재값으로 선택할 수 있음 | 상세 화면의 현재 수위·유속이 오래된 값이 될 수 있다. |
| 필수 | 센서 차트의 24시간·7일 탭, 14:30 기준선, 최고값 문구가 실제 기간 조회·계산을 보장하지 않음 | 발표에서 제공하지 않는 기능으로 오해될 수 있다. |
| 필수 | `riskScore`를 위험 점수로 넓게 표시 | API 경로별 의미가 확정되지 않은 값을 최종 위험도처럼 해석하게 만들 수 있다. |
| 필수 | 센서 카드의 위험·주의 배지가 임계값과 무관하게 고정될 수 있음 | 개별 측정값의 상태를 잘못 전달할 수 있다. |
| 함께 검토 | nullable DTO, API 실패 시 mock fallback, 날짜 포맷 | 통합 환경에서 누락값·장애를 숨기거나 화면 신뢰도를 낮출 수 있다. |

참고로 API DTO와 일부 화면은 이미 `flowVelocityMps`/`m/s`를 사용한다. 이번 작업에서는 남아 있는 `유량`, `m³/min`, `flowRate` 계열 표기까지 실제 코드 검색으로 확인해 일관되게 정리한다.

## 3. 권장 구현 방향

**1차에서는 화면에 잘못된 의미를 전달하는 항목을 한 묶음으로 수정하고, 2차 안정성 항목은 발표 통합 방식이 확정된 경우에만 포함한다.**

| 구분 | 권장 변경 | 이유 |
| --- | --- | --- |
| 단위 | 수위는 `cm`, 유속은 `m/s`, 막힘률만 `%`로 표시 | API 계약과 같고 별도 시설 기준값을 가정하지 않는다. |
| 최신값 | 이력은 `measuredAt` 오름차순으로 정렬하고, 현재값은 `sensorSummary` 우선·정렬된 이력의 마지막 값 fallback으로 선택 | 차트와 현재 카드의 시간 방향을 명시적으로 통일한다. |
| 차트 기간 | 24시간·7일 탭, 가짜 최고값, 14:30 기준선을 제거하고 `최근 측정 데이터` 차트로 표시 | 기간 API가 없는 MVP에서 가장 오해가 적고 발표 화면도 단순해진다. |
| 위험도 | 최종 상태·최종 판단·막힘률·측정 시각을 중심으로 보여 주고, `riskScore` 점수 카드는 제거 | 관리자가 즉시 판단해야 하는 정보는 확정된 `riskLevel`과 `finalDecision`이다. |
| 센서 카드 | 수위·유속·막힘률에는 값과 단위만 표시하고, 상태 배지는 시설의 `riskLevel`에만 사용 | 공식 센서별 임계값이 없는 상태에서 근거 없는 경보를 막는다. |
| 안정성(선택) | DTO null 안전 표시, 실제 API 모드의 명시적 오류 표시, 공통 날짜 포맷 적용 | 발표 통합 중 데이터 누락·서버 장애를 숨기지 않고 일관된 화면을 만든다. |

## 4. 구현 단계

1. `DrainFacility`와 API adapter에서 수위의 UI 표현을 `waterLevelCm` 중심으로 바꾸고, 대시보드·목록·상세 화면의 모든 수위 표시와 진행 바로 사용하는 퍼센트 변환을 제거한다.
2. 센서 이력을 시간순으로 정렬하는 공통 로직을 만들거나 기존 데이터 변환 지점에 명시하고, 상세 화면의 현재 센서값을 `sensorSummary` 우선으로 선택한다.
3. `sensor-trend-chart.tsx`를 현재 API가 제공하는 최근 측정 이력 전용 차트로 정리한다. 기간 탭·임의 샘플링·하드코딩 기준선·가짜 최고값을 제거하고, 라벨을 `현재 수위`와 `현재 유속`으로 통일한다.
4. 상세 페이지와 대시보드/패널에서 `riskScore`를 위험 점수로 표시하는 UI를 제거하거나 최종 상태·판단 문구로 대체한다. 위험도 이력의 점수 노출도 같은 기준으로 점검한다.
5. 센서별 고정 상태 배지를 제거하고, 시설 전체 상태에는 기존 `riskLevel` 색상 체계(good/caution/danger/unknown)를 유지한다.
6. 승인 시 nullable 값의 `-` 처리, mock 모드와 실제 API 오류 상태 분리, 날짜 공통 포맷 적용을 함께 반영한다.
7. `pnpm lint`, `pnpm build`와 대시보드·상세 화면 수동 확인으로 단위, 시간순서, 빈 값, 오류 상태, 모바일 레이아웃을 검증한다.

## 6. 사용자 확인이 필요한 사항

| 항목 | 권장안 | 프로젝트 기준의 이유 |
| --- | --- | --- |
| 작업 묶음 | **1차 필수 항목 6개를 한 번에 진행** | 모두 같은 문제, 즉 API 의미와 다른 발표용 표시를 바로잡는 작업이며 화면·adapter를 함께 바꿔야 일관성이 생긴다. |
| 수위 표시 | **`cm`로 통일하고 수위 퍼센트·진행 바를 제거** | 확정된 API 필드가 `waterLevelCm`이고, 시설별 전체 높이는 MVP 계약에 없다. 막힘률만 %를 유지한다. |
| 현재 센서값 기준 | **`sensorSummary` 우선, 없으면 시간 정렬된 이력의 최신값** | API 명세의 요약값은 현재 상태용이고, 이력은 차트용이라는 역할 분리가 MVP 관제 흐름에 가장 안전하다. |
| 기간 탭 | **24시간·7일 탭과 가짜 기준선/최고값을 제거** | 실제 기간 조회 API가 없는 상황에서 비활성 버튼보다 `최근 측정 데이터` 하나만 보여 주는 편이 발표에서 명확하다. |
| 위험 점수 | **상세·대시보드의 `riskScore` 점수 표시를 제거** | 운영자가 확인할 최종 기준은 `riskLevel`과 `finalDecision`이며, 점수 의미는 API 계약에서 일관되게 확정되지 않았다. |
| 센서 배지 | **수위·유속의 개별 상태 배지를 제거하고 시설 전체 `riskLevel`만 유지** | 센서별 공식 임계값이 없으므로 고정 위험/주의는 잘못된 경보가 될 수 있다. |
| 통합 안정성 항목 | **nullable 처리·실제 API 오류 표시·날짜 공통화를 함께 포함하는 것을 권장** | API 명세에 null/오류 상황이 있고, 발표 통합 단계에서는 mock fallback이 서버 장애를 숨기지 않아야 한다. 단, 환경변수 운용 방식이 바뀔 수 있어 승인 후 진행한다. |

위 권장안이 승인되면 새 패키지나 API 계약 변경 없이 프런트엔드 내부에서 진행한다. 기간 조회 탭을 유지하거나 `riskScore`의 정의를 확정해 표시하고 싶은 경우에는 백엔드 계약 확인이 선행되어야 한다.

## 7. 검증 계획

| 검증 | 기대 결과 |
| --- | --- |
| `pnpm lint` | ESLint 오류 없이 통과 |
| `pnpm build` | Next.js production build 통과 |
| 대시보드·목록·상세 화면 | 수위는 cm, 유속은 m/s, 막힘률은 %로 일관되게 표시 |
| 최신값·차트 | 순서가 섞인 이력에서도 차트는 과거→현재이고 현재 카드는 최신 측정값을 표시 |
| 차트 기능 | 제공되지 않는 24시간·7일 조회, 하드코딩 시간선, 계산 근거 없는 최고값이 없음 |
| 위험도 | 점수 대신 최종 상태와 판단 문구가 표시되고, 센서값에 근거 없는 상태 배지가 없음 |
| 선택 안정성 범위 | null은 `-`로 표시되고, 실제 API 실패가 mock 데이터로 대체되지 않으며 날짜 형식이 통일됨 |
| 반응형 수동 확인 | 360px, 768px, 1280px에서 카드·차트의 단위와 레이블이 겹치지 않음 |

## 8. 리스크와 대응

| 리스크 | 대응 |
| --- | --- |
| 기존 화면이 `waterLevelPct` UI 타입에 넓게 의존 | adapter와 공통 UI 타입부터 변경하고 검색 기반으로 모든 참조를 제거한다. |
| `sensorSummary`와 이력의 시점이 다를 수 있음 | 현재 카드에는 summary 우선, 차트에는 정렬된 이력만 사용해 역할을 분리한다. |
| `riskScore`가 발표 자료에서 필요할 수 있음 | 기본은 숨김이며, 백엔드가 의미와 범위를 확정하면 `예측 신뢰도` 등 정확한 명칭으로 후속 작업한다. |
| 실제 API 오류와 mock 데이터가 혼재 | mock 사용을 명시적인 환경 설정으로 제한하고 실제 API 모드에서는 오류 상태를 보여 준다. |
