## 2. 확인한 맥락과 문제

프로젝트의 관리자 대시보드는 지도, 위험 시설 목록, 상세 정보를 통해 우선 확인 대상을 빠르게 찾는 것이 핵심이다. 대시보드 현황은 이 흐름에서 전체 상황과 위험 시설 수를 즉시 알려 주는 요약 정보이므로 제거하지 않고 유지한다.

현재 `DashboardSummaryBar`는 넓은 화면에서 제목·업데이트 시각 묶음과 5개 수치 카드 묶음을 `justify-between`으로 양끝에 배치한다. 이 때문에 두 묶음 사이가 과도하게 비어 보인다. 또한 상태가 5개인 고정 구조가 2열 또는 3열 그리드로 표시될 때 마지막 행의 남는 칸이 빈 공간으로 남는다.

| 화면 구간 | 현재 문제 | 개선 목표 |
| --- | --- | --- |
| 데스크톱 | 제목과 상태 카드가 양끝으로 분리되어 중앙이 넓고 비어 보임 | 상태 카드가 남은 가로 폭을 활용하도록 해 요약 영역의 정보 밀도를 높임 |
| 모바일 2열 | 다섯 번째 `판단불가` 카드 오른쪽에 빈 칸이 생김 | 마지막 카드가 두 열 전체를 사용하도록 배치 |
| 태블릿 3열 | 네 번째·다섯 번째 카드 뒤에 한 칸이 비어 보임 | 마지막 두 카드를 같은 너비로 한 행에 배치해 빈 칸 제거 |

## 3. 권장 구현 방향

**대시보드 현황은 유지하고, 화면 폭별로 상태 카드의 열 수와 마지막 행의 span을 명시한다.**

| 화면 크기 | 제목/상태 카드 배치 | 상태 카드 배치 | 기대 효과 |
| --- | --- | --- | --- |
| 모바일 (`< 640px`) | 세로 배치 | 기본 2열, 다섯 번째 카드는 2열 전체 span | 좁은 폭에서 수치 카드 크기를 확보하고 마지막 빈 칸 제거 |
| 태블릿 (`640px ~ 1023px`) | 세로 배치 | 6개 가상 열을 사용: 첫 3개는 2칸씩, 마지막 2개는 3칸씩 span | 시각적으로는 균등한 3열 뒤 균등한 2열이 되어 빈 칸 없이 정렬 |
| 데스크톱 (`>= 1024px`) | 제목 영역 + 남은 폭의 상태 카드 영역 | 5열, 각 카드가 같은 폭으로 남은 공간 채움 | 제목과 수치 사이의 과도한 빈 공간을 줄이고 한눈에 보는 현황성을 강화 |

구현 시 실제 수치 카드와 `DashboardSummarySkeleton`이 같은 grid 규칙을 사용하게 한다. 데이터 로딩 중과 완료 후의 카드 위치가 달라지지 않도록 하기 위함이다. 상태 색상은 기존 기준(위험 red, 주의 amber, 양호 green, 판단불가 slate)을 그대로 유지한다.

## 4. 구현 단계

1. `DashboardSummaryBar`의 데스크톱 컨테이너를 제목의 고정/최소 폭 영역과 가변 폭 상태 카드 영역으로 정리한다.
2. 상태 카드 grid에 모바일 2열·태블릿 3열·데스크톱 5열 규칙을 적용한다.
   - 모바일의 마지막 카드에는 전체 행 span을 적용한다.
   - 태블릿의 마지막 두 카드에는 각각 절반 행 span을 적용한다.
3. `DashboardSummarySkeleton`에도 동일한 그리드·span 규칙을 적용한다.
4. 긴 최신 업데이트 시각이 제목 영역을 넘지 않도록 기존 줄바꿈 규칙을 유지하고, 카드의 라벨·수치가 줄바꿈 또는 겹침 없이 보이는지 확인한다.
5. 지도·위험 목록·상세 패널의 데이터 선택과 기존 반응형 전환은 변경하지 않는지 회귀 확인한다.

## 6. 사용자 확인이 필요한 사항

| 항목 | 권장안 | 이유 |
| --- | --- | --- |
| 대시보드 현황 유지 여부 | **유지** | 위험 시설 수와 전체 분포는 지도·목록을 보기 전 우선순위를 빠르게 판단하는 데 필요한 요약 정보다. |
| 데스크톱 카드 폭 | **제목은 필요한 최소 폭만 유지하고, 5개 카드는 남은 폭을 균등하게 사용** | 양끝 정렬로 생기는 중앙 공백을 없애면서도 카드별 수치를 한 줄에 비교하기 쉽다. |
| 모바일 2열의 마지막 카드 | **마지막 카드를 전체 행 너비로 표시** | 홀수 카드 수로 생기는 오른쪽 빈칸을 제거하고, `판단불가` 상태도 다른 상태와 같은 중요도로 보인다. |
| 태블릿 3열의 마지막 행 | **마지막 두 카드를 같은 너비의 2열로 표시** | 5개 상태를 억지로 3열에 유지해 생기는 한 칸의 공백 없이 균형을 맞춘다. |
| 적용 범위 | **현황 요약 바에 한정** | 현재 피드백의 원인을 직접 해결하고 지도·목록·상세 관제 레이아웃의 검증 범위를 불필요하게 넓히지 않는다. |

위 권장안대로 진행하면 별도 디자인 방향 결정 없이 구현할 수 있다. 현황을 제거하거나 상태 항목 수를 줄이는 방향을 원한다면 API 응답·운영 우선순위 표시 기준까지 함께 재검토해야 하므로, 그 경우에는 구현 전에 범위를 다시 정한다.

## 8. 리스크와 대응

| 리스크 | 대응 |
| --- | --- |
| 태블릿 폭에서 최신 업데이트 시각이 길어질 수 있음 | 제목 영역과 카드를 세로 배치하고 날짜 텍스트의 줄바꿈을 허용한다. |
| 구조 선택자가 카드 순서에 의존할 수 있음 | 상태 항목의 순서가 고정인지 확인하고, 읽기 어려운 선택자보다 카드별 span 클래스 또는 작은 배열 메타데이터를 우선 사용한다. |
| 스켈레톤과 실제 카드의 규칙이 달라 레이아웃이 흔들릴 수 있음 | 공통 grid 클래스/구조를 재사용하거나 두 컴포넌트에 동일한 브레이크포인트 규칙을 적용한다. |
| 데스크톱 카드가 지나치게 넓어질 수 있음 | 현재 페이지 최대 폭(`1600px`) 안에서만 확장되도록 유지하고, 카드 내부의 가로 padding은 과도하게 늘리지 않는다. |
