# 11 Dashboard UI 반응형 개선 및 렌더링 최적화 계획

## 1. 작업 개요

| 항목        | 내용                                                                                                           |
| ----------- | -------------------------------------------------------------------------------------------------------------- |
| 현재 브랜치 | `refactor/dashboard-ui-responsiveness`                                                                         |
| 작업 범위   | `/frontend` 내부 대시보드 UI 반응형/가독성 개선 + 선택 변경 시 렌더링 범위 축소                                |
| 최종 목표   | 일반 웹 + 반응형 웹에서 UI 가독성을 높이고, 마커 클릭 시 목록 전체가 아닌 관련 item 중심으로만 갱신되도록 개선 |
| 비목표      | 패키지 추가, 라우팅 변경, 상태관리 라이브러리 교체, 지도 라이브러리 교체                                       |

## 3. 현재 상태 요약 (코드 기준)

1. 대시보드 조합은 `app/page.tsx`에서 수행되고 있으며, `selectedDrainId` 변경 시 page 레벨 컴포넌트가 다시 렌더링된다.
2. 실제 목록 item은 `components/drain-risk-list.tsx` 내부 로컬 `DrainRiskListItem`이 사용 중이다.
3. `components/dashboard/drain-risk-list-item.tsx`는 현재 화면 렌더링 경로에서 실사용되지 않는다.
4. 목록/상세/요약에 표시되는 `updatedAt`은 원본 문자열을 그대로 출력한다.

## 4. 단계별 실행 계획

### 단계 0. Baseline 측정 (코드 변경 전)

목표:

- 현재 리렌더링 범위를 정량적으로 확보한다.

수행:

1. 동일 시나리오로 React DevTools Profiler 3회 측정
   - 시나리오 A: 마커 1회 클릭
   - 시나리오 B: 목록 item 1회 클릭
   - 시나리오 C: 마커 연속 3회 클릭
2. 측정 단위
   - commit 수
   - DashboardPage 렌더 횟수
   - DrainRiskList 렌더 횟수
   - 선택된 item/직전 item 외 item 렌더 여부

산출물:

- step 문서에 baseline 표로 기록
- 캡처 파일은 사용자가 직접 첨부하고, 문서 표에 수치와 코멘트를 반영

캡처 방식(사용자 수행):

1. Chrome에서 대시보드 접속 후 React DevTools Profiler 탭 진입
2. 톱니바퀴 설정에서 `Record why each component rendered while profiling` 활성화
3. 매 시나리오 시작 전 `Clear` 후 `Record` 클릭
4. 시나리오 동작 직후 `Stop` 클릭
5. Commit 뷰에서 최대 렌더 커밋 선택 후 우측 패널 값 확인
   - `DashboardPage` 렌더 횟수
   - `DrainRiskList` 렌더 횟수
   - `DrainRiskListItem` 중 선택/비선택 item 렌더 범위
6. 캡처 저장 규칙
   - 파일명: `baseline-A.png`, `baseline-B.png`, `baseline-C.png`
   - 저장 위치: `frontend/docs/images/`
7. 수치는 step 문서 표에 입력하고, 캡처 링크를 문서에 추가

### 단계 1. 모바일/텍스트/날짜/상세패널 우선순위 개선 (UI 안정화)

목표:

- 화면 높이 압박을 줄이고 정보 스캔 순서를 명확히 한다.

수행:

1. 모바일 세로 길이 완화
   - 지도, 목록, 상세 패널의 `max-h`/`min-h`를 브레이크포인트별로 조정
   - 모바일에서 상세 패널 정보 블록 간격을 축소
2. 긴 텍스트 처리
   - 도로명/주소/시설명/라벨에 `truncate`, `line-clamp`, `break-words`를 위치별 적용
   - 핵심 식별값(ID, 상태)은 항상 보이도록 우선순위 배치
3. 상세 패널 정보 우선순위 재배치
   - 상단: 상태, 판정, 최근 업데이트
   - 중단: 핵심 수치(막힘/수위/유량)
   - 하단: 주소/보조 정보
4. 날짜 포맷 통일
   - 공통 포맷 유틸을 두고 `yyyy-mm-dd hh:MM:ss`로 표시
   - invalid date는 기존 문자열 fallback

완료 기준:

- 모바일(가로 360~430)에서 스크롤 피로 감소
- 긴 텍스트가 레이아웃을 깨지 않음
- 날짜 표기가 대시보드 전반에서 동일

### 단계 2. 위험 목록 item 디자인 정리 + item memoization 검토

목표:

- 목록 시각 구조를 정돈하고 선택 변경 시 item 단위 업데이트를 줄인다.

수행:

1. item 디자인 정리
   - rank, ID, 주소, 상태 배지, 수치 라인 위계 정리
   - hover/selected 대비 강화, 하지만 과도한 색 대비는 지양
2. `DrainRiskListItem` 분리/메모화 검토
   - 현재 로컬 item을 별도 컴포넌트로 분리하거나 유지하되 `React.memo` 적용
   - props를 최소화(`id`, `isSelected`, `isPrevSelected`, 표시용 필드)
   - `onSelect`는 안정된 참조로 전달
3. 정렬 변경/선택 변경 경계 점검
   - 정렬 변경 시 전체 재렌더는 허용
   - 선택 변경 시 가능하면 이전 선택 item + 새 선택 item만 갱신 목표

완료 기준:

- 클릭 시 목록 전체 깜빡임 체감 감소
- 프로파일 상 선택 변경에서 item 렌더 범위가 축소

### 단계 3. selectedDrainId 변경 시 렌더링 범위 재측정 및 경계 보완

목표:

- 마커 클릭 시 갱신 범위를 필요한 영역 중심으로 제한한다.

수행:

1. `app/page.tsx`의 selector/파생값 경계 점검
2. 지도/목록/상세 props 안정화(`useMemo`, `useCallback`, 값 분리)
3. 필요 시 선택 상태 전용 컨테이너 분리
   - 목록 데이터/지도 데이터는 유지
   - 선택 상태 변경은 상세+선택 관련 item 위주 반영

완료 기준:

- 마커 클릭 시 목록의 비선택 item 재렌더가 의미 있게 감소
- 지도/상세의 필요한 갱신은 유지

### 단계 4. 전/후 Profiler 비교 보고

목표:

- 개선 효과를 수치로 확인하고 문서화한다.

수행:

1. 단계 0과 동일 시나리오 재측정
2. 전/후 표 비교
   - commit 수
   - 컴포넌트별 렌더 횟수
   - 상위 컴포넌트 렌더 시간
3. 남은 리스크와 후속 과제 정리

산출물:

- `docs/steps/step-XX-...md`에 Before/After 표와 해석 기록

## 5. 확정 결정 사항 (사용자 확인 반영)

1. 날짜 포맷 정책

- 적용: 추천 방식 채택
- 기준: 로컬 시간대 기준 `yyyy-mm-dd hh:MM:ss` 표시
- 예외: invalid date는 원문 문자열 fallback

2. 모바일 상세 패널

- 적용: 접힘 방식 1차 적용
- 진행 방식: 접힘 UI 적용 후 사용자 확인
- 후속: 유지/롤백 여부를 사용자 확인 후 단계 진행

3. 화면별 텍스트 처리

- 적용: 화면 크기별 추천 방식 채택
- 기준: 모바일 1줄 말줄임, 태블릿 이상 2줄 클램프

4. Profiler 캡처 관리

- 적용: 캡처는 사용자가 직접 수행
- 문서 처리: 본 계획의 캡처 방식을 따라 이미지 첨부 후 문서 업데이트

## 6. 리스크와 대응

1. `selectedDrainId`가 page 상위에서 사용되므로 상위 렌더 자체는 완전 제거가 어렵다.

- 대응: 하위 무거운 목록 item 렌더 범위 축소에 집중.

2. 날짜 포맷 적용 범위가 대시보드 외 페이지까지 넓어질 수 있다.

- 대응: 이번 브랜치는 대시보드 관련 컴포넌트 우선 적용 후, 영향 범위는 별도 정리.

3. 과도한 memoization은 가독성을 떨어뜨릴 수 있다.

- 대응: Profiler에서 이득이 확인된 부분만 제한 적용.

## 10. 단계 종료 공통 루틴 (추가 요청 반영)

각 단계(0~4)가 끝날 때마다 아래를 반드시 수행한다.

1. 변경 파일 점검
- `git status -sb`로 커밋되지 않은 변경 파일 목록 확인
- 필요 시 `git diff -- <파일>`로 핵심 변경 요약

2. 단계 결과 요약
- 이번 단계에서 실제 바뀐 파일과 변경 포인트를 3~5줄로 정리

3. 커밋 메시지 제안 (한글)
- 커밋 제목 1개 제안
- 커밋 본문 bullet 3~5개 제안
- 메시지는 해당 단계의 실제 diff 기준으로 작성

4. 사용자 확인 후 다음 단계 진행
- 사용자가 커밋/보류를 결정하면 다음 단계로 이동

커밋 메시지 형식 가이드:

- 제목: `<type>: <한글 요약>`
- 본문: 변경 이유 + 핵심 변경 + 검증/리스크 순서

type 예시:

- `feat`: 사용자 기능 변화
- `refactor`: 동작 동일 + 구조 개선
- `style`: UI/스타일 중심 변경
- `docs`: 문서 변경
