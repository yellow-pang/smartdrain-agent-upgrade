# SmartDrain Codex Guidelines

SmartDrain은 팀 프로젝트 MVP에서 출발했지만, 현재 저장소는 개인 Fork 기반의 아키텍처 리팩터링과 기능 고도화를 진행하는 작업 공간입니다. Codex는 기존 MVP의 맥락을 존중하되, 새 변경은 개인 고도화 프로젝트의 유지보수성, 실행 재현성, AI 에이전트 확장성을 기준으로 판단합니다.

## 작업 기준

- 작업 전 `git status --short --branch`로 브랜치와 미커밋 변경을 확인합니다.
- 실제 파일 구조, 실행 스크립트, CI 설정을 우선 확인합니다. 문서와 구현이 다르면 현재 실행 가능한 구현을 기준으로 차이를 보고하고 관련 문서를 함께 정리합니다.
- 루트 구조는 `frontend/`, `backend/`, `ai_service/`의 책임 분리를 우선합니다.
- 문서 변경은 현재 구현과 앞으로의 리팩터링 방향을 구분해 적습니다.
- 사용자가 작업 경로나 파일 범위를 지정한 경우 해당 범위 밖의 파일은 수정하지 않습니다.
- 범위 밖 변경이 반드시 필요하면 변경 이유, 대상 파일, 예상 영향을 먼저 보고합니다.
- 요청 해결에 필요하지 않은 전면 재작성, 과도한 공통화, 미래 사용 가능성만을 위한 추상화는 피합니다.

## 변경 범위와 승인 기준

다음 변경은 사용자 승인을 받은 뒤 진행합니다.

- 서비스 간 책임 이동 또는 디렉터리 대량 이동
- API, WebSocket, callback 계약의 호환성을 깨는 변경
- DB schema 또는 Alembic migration 변경
- 패키지 추가, 교체, 삭제
- 환경변수 구조, Docker, Nginx, CI/CD 설정 변경
- 인증, 권한, 비밀값 처리 방식 변경

동일한 책임 범위 안에서의 파일 분리, 이름 정리, 타입 개선, 테스트 추가, 동작을 바꾸지 않는 소규모 리팩터링은 별도 승인 없이 진행할 수 있습니다.

대규모 변경 작업은 수십 개의 파일을 한 번에 옮기지 않고, 파일 손실 방지를 위해 기능 단위나 폴더 단위로 단계를 나누어 진행합니다. 구조 변경이나 대규모 리팩터링이 필요한 경우에는 현재 폴더/파일 목록을 분석하고 변경 파일 목록과 Phase별 이동 계획을 먼저 보고합니다.

## 작업 진행 절차

Codex는 하나의 요청을 브랜치 확인, 계획, 구현, 검증, 작업 기록, PR 문서 작성의 단계로 나누어 진행합니다.

### 1. 요구사항 및 현재 상태 분석

- 요청을 받으면 먼저 작업 목적, 예상 변경 범위, 영향받는 서비스와 파일을 분석합니다.
- 작업을 시작하기 전에 `git status --short --branch`를 실행하여 현재 브랜치와 미커밋 변경을 확인합니다.
- 기존 사용자 변경과 이번 요청에서 발생할 변경을 구분합니다.
- 요구사항이 불명확하더라도 현재 코드와 문서를 조사하여 가능한 범위에서 구체적인 작업 목표를 정리합니다.
- 단순 설명, 코드 분석, 명령어 안내처럼 실제 파일 변경이 없는 요청은 브랜치와 문서 작성 절차를 적용하지 않습니다.

### 2. 브랜치 적합성 확인

- 현재 브랜치 이름과 목적이 요청 내용에 적합한지 확인합니다.
- 현재 브랜치가 다른 작업을 위한 브랜치이거나 요청 목적과 맞지 않으면 코드나 문서를 수정하지 않고 새 브랜치 생성을 안내합니다.
- 이 경우 요청 목적에 맞는 브랜치 이름을 추천합니다.
- 브랜치 이름은 기존 저장소의 브랜치 규칙을 우선하며, 별도 규칙이 없으면 다음 형식을 사용합니다.

```text
feat/<기능명>
fix/<수정대상>
refactor/<개선대상>
test/<테스트대상>
docs/<문서대상>
chore/<설정대상>
```

- Codex는 사용자가 명시적으로 요청하지 않는 한 브랜치를 직접 생성하거나 체크아웃하지 않습니다.
- 사용자가 브랜치를 생성한 뒤에는 `git status --short --branch` 또는 `git branch --show-current`로 올바른 브랜치에 체크아웃되었는지 확인한 후 다음 단계로 진행합니다.
- 브랜치 생성과 체크아웃 안내만 수행한 단계에서는 커밋 메시지를 제안하지 않습니다.

### 3. 계획 문서 작성 및 사용자 승인

다음 조건에 해당하면 실제 구현 전에 `plans` 문서를 작성하고 사용자 승인을 받습니다.

- 사용자 승인이 필요한 변경 범위에 해당하는 경우
- 여러 파일 또는 여러 계층에 걸친 변경인 경우
- 기존 동작, API 계약, 데이터 흐름 또는 실행 방식에 영향을 줄 수 있는 경우
- 구현 방법이 둘 이상이며 선택에 따라 유지보수성이나 확장성이 달라지는 경우
- 사용자가 계획 문서 작성을 요청한 경우

계획 문서에는 최소한 다음 내용을 포함합니다.

1. 요청 배경과 해결하려는 문제
2. 현재 구현 상태와 확인한 근거
3. 작업 범위와 작업 제외 범위
4. 추천 구현 방향과 선택 이유
5. 수정 또는 추가할 예상 파일
6. 단계별 작업 계획
7. 예상 영향과 위험 요소
8. 검증 방법
9. 사용자 승인이 필요한 결정 사항

- 가능한 구현 방향이 여러 개라면 각 방법의 장단점을 비교한 뒤 추천 방향을 명확히 제시합니다.
- 계획 문서는 아직 수행하지 않은 작업을 완료된 것처럼 작성하지 않습니다.
- 계획 문서 작성 후에는 사용자가 문서를 검토할 수 있도록 작업을 멈추고 승인을 기다립니다.
- 사용자가 수정 요청을 하면 계획 문서를 먼저 보완하고 다시 승인을 받습니다.
- 사용자가 계획을 승인하기 전에는 계획 범위를 실제 코드에 반영하지 않습니다.
- 계획 문서 작성으로 Git 변경이 발생한 경우에는 이번 변경만 확인하여 커밋 메시지를 제안합니다.

### 4. 구현 및 검증

- 사용자 승인 후 승인된 계획과 범위에 따라 코드, 설정 또는 문서를 수정합니다.
- 승인된 범위를 넘어서는 변경이 새로 필요해지면 임의로 확대하지 않고 이유, 대상 파일, 영향을 먼저 보고합니다.
- 작업 중 계획과 실제 구현이 달라진 경우 최종 기록에서 그 차이와 변경 이유를 설명합니다.
- 구현이 끝나면 변경 대상 서비스의 검증 스크립트와 CI 설정을 확인하고 영향 범위에 맞는 검증을 실행합니다.
- 검증 실패를 우회하거나 숨기지 않으며, 기존 오류와 이번 작업에서 발생한 오류를 가능한 범위에서 구분합니다.
- 코드와 문서의 내용이 서로 다르면 승인된 작업 범위 안에서 관련 문서를 함께 갱신합니다.
- 구현 및 검증으로 Git 변경이 발생한 경우에는 이번 변경만 확인하여 커밋 메시지를 제안합니다.

### 5. Steps 문서 작성

구현과 검증이 완료되면 사용자가 별도로 요청하지 않아도 해당 작업의 `steps` 문서를 작성합니다. 단, 오탈자 수정이나 한 파일의 매우 작은 변경처럼 별도 작업 기록의 가치가 낮은 경우에는 사용자 요청에 따라 생략할 수 있습니다.

Steps 문서는 다른 팀원이나 이후의 사용자가 기존 대화를 보지 않아도 작업 내용을 이해할 수 있도록 작성합니다.

Steps 문서에는 최소한 다음 내용을 포함합니다.

1. 작업 목적과 배경
2. 변경 전 문제 또는 제약사항
3. 실제로 확인한 코드와 구조
4. 적용한 해결 방법
5. 해당 방법을 선택한 이유
6. 수정한 주요 파일과 각 파일의 역할
7. 주요 데이터 흐름 또는 실행 흐름
8. 수행한 검증과 결과
9. 기존 계획과 달라진 내용 및 이유
10. 남은 제한사항이나 후속 작업

- 단순히 “에이전트가 파일을 수정했다”는 형태로 작성하지 않습니다.
- 어떤 문제를 확인했고, 왜 해당 구조나 구현 방식을 선택했으며, 변경 결과가 시스템에 어떤 영향을 주는지 설명합니다.
- 코드 전체를 나열하기보다 핵심 변경과 판단 근거를 중심으로 작성합니다.
- 흐름 설명이 필요한 경우 Mermaid 흐름도, 표 또는 단계별 목록을 사용할 수 있습니다.
- 아직 수행하지 않은 작업이나 확인하지 않은 결과를 완료된 것처럼 기록하지 않습니다.
- Steps 문서 작성으로 Git 변경이 발생한 경우에는 이번 변경만 확인하여 커밋 메시지를 제안합니다.

### 6. 사용자 변경 확인

- 구현, 검증, Steps 문서 작성이 끝나면 사용자가 변경 내용을 확인할 수 있도록 최종 결과를 보고합니다.
- 사용자가 추가 수정 요청을 하면 현재 브랜치와 기존 계획의 범위 안에서 수정하고 Steps 문서도 실제 결과에 맞게 갱신합니다.
- 추가 요청이 기존 작업 목적을 벗어나는 별도 기능이라면 현재 작업에 섞지 않고 새 브랜치와 새 작업으로 분리할 것을 안내합니다.
- 각 수정 요청이 완료될 때마다 현재 Git 변경을 확인하고 해당 변경에 맞는 커밋 메시지를 제안합니다.

### 7. PR 문서 작성

- PR 문서는 사용자가 변경 내용을 확인한 뒤 명시적으로 요청한 경우에 작성합니다.
- 기존 `pr` 문서가 있으면 파일 위치, 제목 구조, 섹션 순서와 표현 방식을 먼저 확인하고 동일한 형식을 우선합니다.
- 기존 양식을 유지하되 작업 특성상 필요한 경우 표, 체크리스트, 코드 블록 또는 Mermaid 흐름도를 추가할 수 있습니다.
- PR 문서는 Plans와 Steps 문서의 내용을 그대로 반복하지 않고, 리뷰어가 변경 목적, 핵심 구현, 영향 범위와 검증 결과를 빠르게 파악할 수 있도록 요약합니다.

PR 문서에는 원칙적으로 다음 내용을 포함합니다.

1. 작업 배경과 목적
2. 주요 변경 사항
3. 영향받는 범위
4. 구현 또는 데이터 흐름
5. 검증 결과
6. 리뷰 시 확인할 부분
7. 관련 Plans 및 Steps 문서
8. 알려진 제한사항 또는 후속 작업

- 실제 Git 변경과 Plans, Steps 문서를 확인한 뒤 작성합니다.
- 구현되지 않은 내용이나 실행하지 않은 검증을 포함하지 않습니다.
- PR 문서 작성으로 Git 변경이 발생한 경우에는 이번 변경만 확인하여 커밋 메시지를 제안합니다.

### 8. 작업 종료

- 실제 Pull Request 생성, 병합, 브랜치 삭제는 사용자가 직접 수행합니다.
- 사용자가 PR을 생성하고 병합한 뒤 브랜치를 삭제하면 해당 요청의 작업 흐름은 종료된 것으로 봅니다.
- 새로운 기능이나 수정 요청은 가능하면 새 브랜치와 새 대화에서 진행합니다.
- 이전 작업의 후속 요청이라도 목적과 변경 범위가 달라지면 별도 작업으로 분리합니다.

## 작업 문서 위치와 이름

- Plans, Steps, PR 문서는 저장소의 기존 문서 구조와 파일 이름 규칙을 먼저 확인하고 일관되게 작성합니다.
- 기존 규칙이 없으면 전체 저장소 작업은 `docs/`, 특정 서비스에 한정된 작업은 해당 서비스의 `docs/` 아래에 작성합니다.
- 문서 유형별 기본 디렉터리는 다음과 같습니다.

```text
docs/plans/
docs/steps/
docs/pr/
```

- 파일 이름은 기존 문서의 번호 체계와 명명 규칙을 우선합니다.
- 기존 규칙이 없으면 다음 형식을 사용합니다.

```text
plan-<번호>-<작업명>.md
step-<번호>-<작업명>.md
pr-<번호>-<작업명>.md
```

- 하나의 작업에 속하는 Plans, Steps, PR 문서는 가능한 한 동일한 번호와 작업명을 사용하여 연관성을 유지합니다.
- PR 문서에는 관련 Plans와 Steps 문서의 상대 경로를 포함합니다.

## 프로젝트 구조 책임

| 경로                    | 책임                                                                 |
| ----------------------- | -------------------------------------------------------------------- |
| `frontend/`             | Next.js 관리자 대시보드, 지도/차트/상세 화면, 브라우저 공개 환경변수 |
| `backend/`              | FastAPI API 서버, DB 모델, Alembic migration, WebSocket, AI callback |
| `ai_service/`           | 비동기 AI 분석 서버, YOLO/OpenCV, XGBoost, Backend callback          |
| `ai-vision/`            | 모델 학습, PoC, 실험 코드와 산출물                                   |
| `mock_data/`            | 로컬/시연용 샘플 이미지와 데이터                                     |
| `nginx/`                | same-origin reverse proxy 설정                                       |
| `docs/`                 | 목적별 프로젝트 문서. `legacy-mvp/`는 기존 MVP 산출물, `reference/`는 현재 참조 기준, `verification/`은 검증·운영 기준, `plans/steps/pr`은 신규 작업 이력 |
| `.jenkins/`, `jenkins/` | CI/CD와 배포 보조 파일                                               |

## Frontend 작업

- Next.js App Router, TypeScript, React 함수형 컴포넌트를 사용합니다.
- `frontend/package.json`의 scripts를 확인한 뒤 영향 범위에 따라 `pnpm lint`, `pnpm exec tsc --noEmit`, `pnpm build`를 실행합니다.
- 컴포넌트 파일은 lowercase kebab-case, export 이름은 PascalCase를 사용합니다.
- Tailwind CSS와 `components/ui/`의 기존 primitive를 우선 사용합니다.
- API base URL은 same-origin이면 비워 두고, 직접 backend를 호출하는 로컬 개발에서만 `NEXT_PUBLIC_API_BASE_URL`을 사용합니다.

## Backend 작업

- backend 단독 실행 기준 파일은 `backend/` 아래에 둡니다.
- Alembic 설정은 `backend/alembic.ini`, migration 파일은 `backend/alembic/`을 기준으로 합니다.
- 런타임 의존성은 `backend/requirements.txt`를 기준으로 관리합니다.
- REST, WebSocket, callback 계약 변경 시 producer와 consumer 양쪽의 타입, adapter, 문서를 함께 점검합니다.
- DB 변경 시 migration 생성 여부와 기존 데이터에 미치는 영향을 확인합니다.

## AI Service 작업

- AI 서버 런타임 의존성은 `ai_service/requirements.txt`를 기준으로 관리합니다.
- YOLO/XGBoost 모델 계층은 HTTP callback 정책을 알지 않도록 유지합니다.
- Backend callback, timeout, retry, HTTP 오류 매핑은 `ai_service/http` 계층에서 다룹니다.
- 모델 파일이 없는 환경에서도 가능한 단위 테스트를 우선합니다.
- 실제 모델 파일과 비밀값은 Git에 커밋하지 않습니다.

## 의존성과 생성 파일

- 새 의존성을 추가하기 전에 기존 의존성이나 표준 기능으로 해결 가능한지 확인합니다.
- 패키지 변경 전에는 추가 목적, 대안, 번들 또는 런타임 영향을 보고합니다.
- lockfile은 실제 패키지 변경이 발생한 경우에만 수정합니다.
- `.next/`, `node_modules/`, `coverage/`, `__pycache__/`, 로그, 임시 파일, 모델 가중치 등 생성 산출물은 Git 추적 대상으로 추가하지 않습니다.
- 모델 파일이나 대용량 파일을 생성하기 전에 저장 위치와 `.gitignore` 적용 여부를 확인합니다.

## 문서 작성 기준

- 루트 `README.md`는 포트폴리오용 대표 문서이자 전체 실행 안내입니다.
- 하위 README는 각 서비스 담당자가 바로 실행/검증할 수 있는 운영 노트로 유지합니다.
- 오래된 팀 프로젝트 표현, 절대 로컬 경로, 현재 구조와 다른 명령어는 발견 시 함께 정리합니다.
- 문서는 기본적으로 한국어로 작성하고, 명령어는 Windows PowerShell과 Docker Compose 기준을 우선 안내합니다.

## 테스트와 검증

- 검증 명령을 실행하기 전에 각 서비스의 `package.json`, `requirements.txt`, 테스트 설정, CI 설정을 확인합니다.
- 저장소에 실제로 구성된 검증 도구만 사용하며, 검증 도구를 추가해야 할 경우에는 먼저 승인을 받습니다.
- 파일 이동 후에는 관련 import 경로가 깨지지 않았는지 TypeScript 컴파일, Python import 검사 등 최소 검증을 실행합니다.
- Backend와 AI Service 변경은 사용 가능한 경우 Python import 검사와 관련 `pytest`를 실행합니다.
- API, WebSocket, callback 변경 시 producer와 consumer 양쪽의 타입, adapter, 문서를 함께 점검합니다.
- 버그 수정 시 가능하면 문제를 재현하는 테스트를 추가합니다.
- 외부 DB, 네트워크, 모델 파일 없이 실행할 수 있는 테스트를 우선합니다.
- 검증 실패를 숨기거나 타입 검사 및 빌드 검사를 우회하는 설정을 추가하지 않습니다.
- 검증을 실행하지 못한 경우 실행하지 못한 항목과 이유를 최종 보고에 명시합니다.
- 기존 오류와 이번 변경으로 발생한 오류를 가능한 범위에서 구분합니다.

## 안전한 파일 및 Git 작업

- 작업 시작 시 확인한 기존 변경과 이번 작업에서 만든 변경을 구분합니다.
- 사용자 변경으로 판단되는 내용을 덮어쓰거나 되돌리지 않습니다.
- 실제 `git commit`, `git push`, 브랜치 삭제, 태그 생성은 사용자가 직접 수행합니다.
- 다음과 같은 파괴적 명령은 사용자 승인 없이 실행하지 않습니다.
  - `git reset --hard`
  - `git clean -fd`
  - `git restore .`
  - `git checkout -- <file>`
  - `rm -rf`
  - `docker compose down -v`
  - `docker system prune`

## 환경변수 및 보안 기준

- API 키, 데이터베이스 접속 정보, 모델 비밀값 등 모든 민감한 정보는 소스 코드에 하드코딩하지 않습니다.
- 환경변수는 각 서비스 폴더(`backend/`, `ai_service/` 등)의 `.env.example` 파일을 기준으로 관리하며, 실제 값은 로컬 `.env`에 숨겨 처리하도록 유도합니다.
- 새로운 환경변수가 필요해진 경우, 실제 값을 커밋하는 대신 `.env.example` 문서만 업데이트하고 사용자에게 로컬 설정을 안내합니다.

## 최종 보고와 커밋 메시지 제안

작업 완료 보고에는 다음 내용을 포함합니다.

1. 변경 목적과 결과
2. 수정하거나 이동한 주요 파일
3. 실행한 검증과 결과
4. 실행하지 못한 검증 또는 남은 위험
5. 사용자 확인이 필요한 사항

브랜치 이름 추천과 브랜치 체크아웃 확인 단계에서는 커밋 메시지를 제안하지 않습니다.

Plans 문서 작성, 구현, Steps 문서 작성, 추가 수정, PR 문서 작성 등 실제 Git 변경이 발생한 각 작업 단계가 끝나면 다음 절차를 수행합니다.

1. `git status --short`와 필요한 범위의 `git diff`를 확인합니다.
2. 작업 시작 전부터 존재한 사용자 변경과 이번 단계에서 발생한 변경을 구분합니다.
3. 이번 작업 단계에서 발생한 변경만 기준으로 한글 Conventional Commit 메시지를 제안합니다.
4. 서로 목적이 다른 변경이 섞여 있으면 하나의 커밋으로 묶지 않고 커밋 분리 방향을 제안합니다.
5. 실제 `git add`, `git commit`, `git push`는 사용자가 직접 수행합니다.

```text
refactor(frontend): 대시보드 컴포넌트 책임 분리

- 긴 페이지 컴포넌트의 유지보수성 개선
- 지도와 위험 목록 컴포넌트 분리
- 화면 동작과 API 계약은 기존과 동일하게 유지
```
